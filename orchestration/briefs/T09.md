# 子任务简报：T09

## 任务名称

小红书评论采集。

## 任务目标

在 T08 小红书详情采集完成并验收后，实现阶段 1 的评论与回复采集链路。

本任务必须实现：

- 评论任务执行入口
- 一级评论采集
- 二级回复采集或在现有接口边界内保存父子关系
- 评论分页游标保存
- 评论 L2 入库
- 原始评论页快照保存
- 使用 `xhs` 平台标识运行的可测试评论采集链路

本任务只实现评论和回复采集，不实现评论区动态预算、不实现高价值内容池、不实现完整断点续传策略、不修改数据库模型或迁移。

## 输入

- T08 已保证目标内容存在于 `contents`。
- 评论任务使用 `task_type="comments"` 或语义等价的任务类型。
- 评论任务的 `target_id` 应指向平台内容 ID，或在 `payload_json` 中明确提供平台内容 ID。
- `PlatformAdapter.list_comments(platform_content_id, cursor, limit)` 返回统一 `CommentPage`。
- 如 Adapter 已区分一级评论和回复，应按 `parent_platform_comment_id` 写入父子关系；如尚未提供独立回复接口，必须至少覆盖同一 `CommentPage` 中的父子评论关系。

## 输出

完成后应具备：

- 一个 worker 层评论采集入口，可处理一条评论任务。
- 评论采集入口能读取任务目标内容 ID，调用传入的 `PlatformAdapter.list_comments`。
- 评论结果作为 L2 数据入库：
  - 写入或更新 `comments`
  - 写入 `parent_comment_id` 形成一级评论和二级回复关系
  - 写入或更新评论者 `public_profiles`
- 原始评论页快照保存：
  - 将原始评论页数据序列化为 JSON 文件或等价本地快照
  - 写入 `Snapshot(entity_type="content", entity_id=content.id, snapshot_type="comments_page", object_storage_path=..., content_hash=...)`
- 评论分页游标保存到 `CollectionTask.cursor_json`，至少包含：
  - `next_cursor`
  - `has_more`
  - `limit`
  - `platform_content_id`
- 任务状态更新：
  - 没有后续页时标记为 `completed`
  - 有后续页时可标记为 `partial` 并保留游标
  - 异常时通过状态机进入 retry/failed，而不是吞掉错误
- 测试使用 `MockPlatformAdapter`，不依赖真实小红书网络。
- 如果当前环境无法验证真实 Chrome 登录态，必须在报告“未完成事项/风险与问题”中明确说明；不得把 Mock 测试描述成真实页面验证。
- `orchestration/reports/T09.md`。

## 验收条件

1. 可处理一条评论采集任务，并调用 Adapter 获取评论页。
2. 一级评论写入 `comments`，重复执行不产生重复评论。
3. 二级回复能通过 `parent_comment_id` 建立父子关系。
4. 评论者公开 ID 能映射或创建 `public_profiles` 身份记录。
5. 评论分页游标保存到任务 `cursor_json`。
6. 同一次评论页保存一条可追溯原始快照记录，快照内容哈希稳定。
7. 无后续页时任务进入 `completed`；有后续页时任务进入 `partial` 并保留游标。
8. Adapter 或入库异常时任务通过状态机进入 retry/failed，并记录错误。
9. 测试覆盖成功页、分页页、重复执行幂等、回复关系、异常重试和快照写入。
10. 不修改数据库模型、迁移、查询 API、调度状态机和全局进度文件。
