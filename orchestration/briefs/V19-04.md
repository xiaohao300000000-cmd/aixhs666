# 子任务简报：V19-04

## 任务名称

妙搭今日行动、任务结果、连续审核、客户中心和系统健康产品闭环。

## 任务目标

把已经完成的 V19-01 统一客户推进、V19-02 Base CRM、V19-03 人类报告与每日审核队列，真正呈现在妙搭主操作台中。用户打开线上应用后应在 10 秒内知道今天最重要的事，能从 Run 结果进入候选、连续审核、进入客户详情、查看 Base 去向和系统阻塞，而不需要理解原始 JSON、内部 task type、数据库状态名或异常堆栈。

V19-04 不实现公开回复草稿/确认/真实发送，也不实现 14:00、15:00、21:00 调度或回复检查。尚未实现的能力必须显示为明确的“尚未启用/后续阶段”，不能用假按钮、演示数据或假成功文案冒充。

## 必须实现

### 1. 同源 BFF 资源边界

在现有 Operator controller/service 风格中只做代理，不复制 Python 编排。至少覆盖：

- `GET /api/operator/review-queue`：业务日、分层、offset、limit；
- `POST /api/operator/review-queue/continue`：additional、priority_only、业务日、幂等键；
- `GET /api/operator/tasks/runs/:runId/report`；
- `GET /api/operator/tasks/runs/:runId/candidates`：支持 layer；
- `GET /api/operator/customers`；
- `GET /api/operator/customers/:customerId`；
- `GET /api/operator/customers/:customerId/timeline`。

所有路径继续自动附加服务端 Bearer token；不得把 token、后端根地址或原始 Axios 错误返回浏览器。FastAPI 的 400/401/404/422/503 必须翻译为可区分、可行动的安全错误，而不是全部折叠为“后端不可达”。写请求必须保留调用方提供的幂等键，网络重试不得生成新 key。

### 2. 今日工作台

首页从“工程监控”改为“今天做什么”，默认优先级：阻塞业务的异常、已确认待发送（本阶段只显示尚未启用）、客户新回复（本阶段只显示尚未启用）、到期跟进、高优先级审核、普通审核、不确定/QC、非阻塞异常。

页面至少展示：

- 今日最重要的事和唯一主按钮；
- 今日审核 `completed / target`、pending、QC；
- 高优先级/普通/不确定审核入口；
- 客户数量、待首次联系、已联系待回复、到期跟进等当前真实可计算项；无法由现有 API 可靠计算的指标显示“尚未接入”，不得显示 0 冒充；
- 最近完成 Run 的一句话业务结论和“查看结果”；
- Base CRM 入口；
- 系统健康摘要和“查看详情”；
- 最后更新时间、手动刷新和明确自动刷新状态。

Worker、task type、原始状态和 JSON 只进入 `/system-health` 或折叠技术详情。后端不可达时不得渲染静态演示数字。

### 3. 任务中心与人类结果

保留现有创建/预览/排队/取消/重试/复制能力，但业务默认视图必须：

- 模板使用业务语言；默认强调“扩大潜在线索池”；
- 运行前说明 Campaign、范围、数量、是否访问平台、是否外部写入、不会自动联系客户；
- Run 完成后显示 `business_report_json` 的一句话结论、业务漏斗、排除原因、数据去向和下一步；
- 漏斗数字可按 layer 钻取候选明细；
- “审核本次候选”深链保留 `run_id`，落到 `/leads` 对应批次；
- 原始 result、events、checkpoint 引用和错误细节默认折叠为“技术详情”；
- 没有报告时显示“该历史任务尚未生成业务报告”，不得退回只显示 `succeeded` 或原始 JSON。

### 4. 连续审核

`/leads` 默认读取 V19-03 持久日队列，不再仅展示旧 Lead pending 数组。必须：

- URL 保留 `queue_date`、`run_id`、`layer`、`candidate_key` 或当前位置，刷新/返回不丢批次；
- 显示 `completed / target / pending / quality_control`；
- 左侧稳定队列、中间证据、右侧动作后果预览，底部固定“推进为客户 / 暂缓判断 / 淘汰线索”；
- 审核成功后具体说明客户 ID、CRM 阶段、下一步、Base 状态，并自动选择下一条 pending；
- 同一提交期间禁用所有动作；重复点击复用同一幂等键；失败后允许用原 key 重试；
- 支持“继续审核 20 条”和“只看高优先级”，同一点击重试不得重复扩展；
- 队列项缺少 `lead_id` 时不可调用错误目标，显示明确的不可审核原因和技术详情；
- 已完成、空队列、全部完成、后端不可达、部分字段缺失各有独立状态；
- 暂缓必须要求原因和恢复日期，淘汰必须要求结构化原因。

本阶段不得生成或发送公开回复；推进后只显示“公开回复草稿功能将在 V19-05 开放”。

### 5. 客户中心

新增 `/customers` 和 `/customers/:id`：

- 客户列表使用真实 `qualified` 客户，展示客户名、CRM 阶段、下一步、同步状态、更新时间和 Base 记录入口；
- 客户详情展示摘要、地区/需求/课程、意向、当前行动、下次跟进、最近联系、证据、AI 判断、同步状态、Base 深链和妙搭稳定 URL；
- 时间线按时间顺序翻译发现、人工审核、跟进和同步事实，原始 `event_type`/`data` 默认折叠；
- 没有 Base mapping、同步失败、没有时间线、客户不存在和后端不可达均有明确文案；
- 联系草稿、发送状态、飞书提醒若尚无真实事实，显示后续阶段边界，不制造假记录；
- 页面之间的 Run、候选、客户和 Base 链接必须落到具体对象，不只打开首页。

### 6. 系统健康

新增 `/system-health`，承接首页下沉的工程信息：

- Operator 后端连接与最后成功时间；
- Worker healthy/stale、当前任务和最后心跳；
- 失败任务及安全错误摘要；
- Base/飞书状态只显示现有 API 可证明的事实；不可证明时显示“未提供状态”；
- 提供刷新和回到对应业务动作的链接；
- 不展示 token、内部 URL、完整堆栈、完整个人数据。

阻塞业务的异常在首页提升优先级；非阻塞异常只在系统健康页展开。

### 7. 路由、导航与状态恢复

`AppShell` 和路由至少包含：

- `/` 今日工作台；
- `/tasks` 业务任务与结果；
- `/leads` 连续审核；
- `/customers` 客户中心；
- `/customers/:id` 客户详情；
- `/system-health` 系统健康。

导航不得继续把已实现页面标为“即将开放”。窄屏必须可以完成主动作；桌面端不允许横向溢出或被固定按钮遮挡。优先复用现有组件和 Tailwind 设计语言，不进行与 V19-04 无关的全站换肤。
