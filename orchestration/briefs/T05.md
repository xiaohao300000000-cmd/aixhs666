# 子任务简报：T05

## 任务名称

去重与发现关系。

## 任务目标

在 T02 核心数据模型和 T04 平台采集对象基础上，实现阶段 0 的存储去重与发现关系写入能力。

本任务必须实现：

- 内容去重
- 评论去重
- 多查询发现关系
- 文本哈希

本任务只实现可测试的存储层服务和文本哈希工具，不实现真实平台采集、不实现 API、不实现后台 worker、不修改数据库模型或迁移。

## 输入

- T02 已提供唯一键：
  - `contents(platform, platform_content_id)`
  - `comments(platform, platform_comment_id)`
  - `public_profiles(platform, platform_user_id)`
- T04 已提供统一采集对象和确定性 Mock Adapter。
- `discovery_relations` 记录查询如何发现内容：
  - `query_id`
  - `content_id`
  - `rank_position`
  - `result_page`
  - `discovery_method`
  - `discovered_at`

## 输出

完成后应具备：

- 一个存储层 ingest/dedup 服务，可接收 T04 的统一采集对象并写入 ORM 模型。
- 内容按 `platform + platform_content_id` 去重，重复内容更新可变字段和 `last_seen_at`，不新增内容行。
- 评论按 `platform + platform_comment_id` 去重，重复评论更新可变字段和 `last_seen_at`，不新增评论行。
- 用户资料按 `platform + platform_user_id` 去重，重复用户更新资料和 `last_seen_at`，不新增用户行。
- 同一内容被多个查询发现时，可产生多条 `discovery_relations`，每条关系指向对应 query 和同一 content。
- 同一 query 和同一 content 的重复发现应保持幂等，避免无意义重复关系。
- 一个稳定文本哈希工具，用于对规范化文本生成一致哈希，供后续文本处理任务复用。
- `orchestration/reports/T05.md`。

## 验收条件

1. 同一内容重复 ingest 后，`contents` 中只保留一行，并更新可变字段和 `last_seen_at`。
2. 同一评论重复 ingest 后，`comments` 中只保留一行，并更新可变字段和 `last_seen_at`。
3. 同一公开用户重复 ingest 后，`public_profiles` 中只保留一行，并更新资料和 `last_seen_at`。
4. 同一内容被两个不同查询发现时，产生两条 `discovery_relations`，但 `contents` 仍只有一行。
5. 同一 query/content 的重复发现不重复写入关系。
6. 文本哈希对空白、大小写等基础差异有稳定规范化结果；不同文本产生不同哈希。
7. 测试使用现有 ORM 模型和 T04 Mock 数据，不依赖网络。
8. 不修改数据库模型、迁移、采集器接口和全局进度文件。
