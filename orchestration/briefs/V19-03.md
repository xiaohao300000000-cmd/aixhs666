# 子任务简报：V19-03

## 任务名称

Skill Run 人类业务报告、高召回分层队列和每日 50 条审核计划。

## 任务目标

把现有 Skill Run 的原始计数/JSON 升级为稳定、可钻取、可审计的人类业务报告，并把真实未审核候选按高召回原则生成确定性的每日审核队列。PostgreSQL 继续作为唯一事实源；每日 50 条是展示与行动预算，不是保存上限，不得删除第 51 条以后的候选。

本任务只完成 V19-03 后端事实、Operator API 和飞书任务结果文案。不得提前修改妙搭页面、实现 V19-04 客户中心/工作台，或实现公开回复发送和定时调度。

## 已确认业务规则

- 结果页首先回答：发生了什么、为什么、结果去了哪里、下一步做什么；`succeeded` 和原始 JSON 只能进入技术详情。
- 高召回优先：信息不足、地区未知、意向弱、低置信度、单条证据或课程不明确不得单独触发自动排除。
- 自动排除只允许高确定性原因：明确广告/机构账号、明确无关、完全重复、已删除且无有效证据、明确违反硬条件、测试数据、人工确认无效且没有新证据。
- 同一客户的多条证据合并为一个审核位置；优先使用 `public_profile_id` 形成稳定候选键，无 profile 时才回退到来源实体键。
- 每日默认 50 条：5 条质量控制保留位 + 45 条业务优先级填充位。
- 质量控制位依次使用不确定候选、自动排除抽检、普通候选回填。
- 其余 45 条按高优先级、普通、不确定排序填充；高优先级没有 30 条上限。
- 强信号可作为 `emergency` 插队并让队列超过 50，但必须显式标记；普通重算不得打乱已创建队列。
- 重跑必须幂等；同一天同一候选不得重复入队，已经人工审核的候选不得重新进入待审核队列。
- 审核动作完成后，对应队列位置必须反映完成状态和今日进度；不得由前端临时推算。
- 原始 Skill Run summary、checkpoint、事件和模型结果必须保留，不能为了人类报告覆盖或删除审计事实。

## 必须实现

### 1. 持久审核队列事实

新增最小、命名明确的每日审核队列模型与 `0019_review_queue` 迁移。至少保存：队列日期（Asia/Shanghai 业务日）、稳定候选键、代表 Screening/Lead/Profile、来源 Run、来源 Screening 集、分层、槽位类型、确定性优先级、位置、状态、紧急标记、入队原因、排除抽检原因、人工决定和审计时间。

必须有：

- 同一天同一候选唯一约束；
- 按 `(queue_date, status, position)` 的读取索引；
- 同一 profile 多证据/多 Screening 合并；
- 已审核候选不重新进入 pending；
- 不为纯改名复制现有 Lead/Screening/Run 表。

### 2. 高召回分层

新增聚焦、纯业务分类服务，输出：

```text
priority_review
standard_review
uncertain_review
automatic_exclusion
```

必须明确区分硬排除证据与软不确定信号。`review_status=rejected`、`valuable=false`、低意向、低置信度、`model_uncertain`、`location_unknown` 或资料不足本身都不得直接成为自动排除的充分条件。原始 AI/Campaign 判断继续保留在技术详情和候选证据中。

排序必须确定性稳定，至少以分层、意向、置信度、更新时间和稳定 ID 形成完整 tie-break；重复构建不能因普通分数变化重排当天已有位置。

### 3. 人类业务报告

为每个完成的 Skill Run 生成并持久保存结构化业务报告，同时保留原有 summary keys、checkpoint 和事件。报告至少包含：

- 一句话业务结论；
- 分析范围与处理数量；
- 高优先级、普通、不确定、明确自动排除数量；
- 排除原因统计；
- 本次候选明细与证据摘要；
- 今日队列准备数量、质量控制数量、紧急数量；
- PostgreSQL、妙搭审核入口、Base/飞书同步的去向与状态；
- 下一步主动作“审核本次候选”；
- 默认折叠的技术详情引用，不复制或泄漏密钥、完整异常堆栈、完整隐私数据。

没有候选时必须使用清晰业务文案，不能显示假成功或虚构 50 条。

正常 Skill Run `summarize` 阶段必须调用同一报告/队列服务；另提供对历史 succeeded Run 的幂等重建入口，供迁移与真实验收使用。

### 4. 每日 50 条计划

实现聚焦服务 `services/daily_review_queue.py`：

- 默认预算 50；先确定 5 个质量控制位，再填 45 个业务位；
- QC 顺序为不确定、自动排除抽检、普通回填；
- 业务位按高优先级、普通、不确定填充，不给高优先级设上限；
- 第 51 条以后继续保留为候选 backlog，不删除、不篡改 Screening；
- 支持幂等“继续审核 20 条”和“只看高优先级”的后端查询/扩展能力；
- 支持显式 emergency 追加并超过 50；
- 单个坏候选不能让整个计划静默失败，错误必须可见；
- 计划创建与外部 Base、飞书、小红书动作解耦。

### 5. 审核进度闭环

V19-01 的统一 `promote/defer/reject` 成功后，匹配的 pending 队列位置必须幂等更新为已完成并记录决定；重复动作不得重复增加完成数。今日进度由持久队列事实计算，返回 `completed/target/pending/quality_control`，不能只数当前页面数组。

### 6. Operator API 与飞书结果

在现有 Operator API 风格下至少提供：

- Run 人类报告读取；
- Run 候选/排除明细按层级钻取；
- 按 Run/业务日幂等准备审核队列；
- 今日队列与进度读取；
- “继续 20 条”和“只看高优先级”的稳定后端能力。

写接口必须要求非空幂等键。响应返回稳定 `run_id`、`candidate_key`、`lead_id`/`screening_id`、分层、原因、证据、队列位置、状态、妙搭相对深链和明确下一步。公网网关继续只开放受 token 保护的 Operator router，不开放管理员、飞书回调或原始数据库接口。

飞书任务完成卡/结果卡使用同一业务报告摘要，主文案不得退回原始 JSON 或仅显示 `succeeded`；卡片生成测试不得真实发送消息。
