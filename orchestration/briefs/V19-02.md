# 子任务简报：V19-02

## 任务名称

Base 客户 CRM、跟进记录、允许字段双向同步与妙搭深链。

## 任务目标

在不改变“PostgreSQL 是唯一事实源”的前提下，把现有客户级飞书表升级为可用的个人 CRM，并新增一事一行的跟进记录投影。妙搭、飞书卡片和 Base 必须引用同一个后端客户 ID；Base 只允许白名单人工字段回写，系统事实字段不得覆盖后端。

本任务只完成 V19-02。不得提前实现每日 50 条、妙搭客户中心、公开回复发送或定时调度。

## 已确认业务规则

- PostgreSQL/Python 后端是唯一事实源。
- Base 是可重建的 CRM 投影，不承载核心状态机。
- 一位客户一行；一次联系、回复或人工操作一行。
- Base 可写白名单：`CRM阶段`、`下次跟进时间`、`跟进备注`、`联系结果`、`客户标签`。
- 系统只读字段：后端客户 ID、AI 判断、原始证据、来源链接、Campaign、运行批次、发送状态/证据、同步版本。
- 同步失败只重试同步，不重放客户推进、外发或其他业务动作。
- 对真实 Base 只允许读取、添加字段/表/视图、幂等 upsert 和可逆测试值更新；禁止删除字段、删除表、删除记录或覆盖未知数据。

## 必须实现

### 1. 客户 CRM 事实字段

为现有 Lead 增加满足 CRM 投影所需、命名明确的最小字段；必须包含稳定 CRM 阶段、客户标签、最近联系时间/结果、单调同步版本。不得复制一张仅为改名存在的 Customer 表。

### 2. 跟进记录事实

新增 `CustomerFollowupRecord`：至少包含后端记录 ID、客户 ID、唯一事件键、发生时间、操作类型、渠道、目标、内容、客户回复、本次结果、下一步、下次跟进时间、来源入口、平台证据、完成状态和审计时间。`event_key` 唯一；按 `(lead_id, occurred_at)` 建索引。

### 3. 推进客户后果

V19-01 的 `promote` 必须幂等创建“待首次联系”跟进事实，并返回 CRM 同步为待处理的明确后果；不得在数据库事务中直接调用飞书网络。

### 4. Base 投影

实现聚焦服务 `services/customer_crm_sync.py`：

- 客户 CRM 一客户一行；
- 跟进记录一事实一行；
- 复用 `FeishuBitableRecord` 保存两类映射；
- 创建结果不确定时进入 reconciliation 状态，禁止盲目重复创建；
- 单条失败不阻塞其他记录；
- 重跑幂等；
- 不把密钥、内部异常堆栈或完整隐私数据写入报告。

### 5. 白名单回写

实现增量 pull，只接受已确认的五类人工字段。必须验证同步版本/远端更新时间，保护系统字段和较新的后端事实。人工阶段变化写时间线与跟进记录；重复拉取不得重复生成事件。

### 6. Operator API 与深链

至少提供：

- `GET /operator/api/customers`
- `GET /operator/api/customers/{customer_id}`
- `GET /operator/api/customers/{customer_id}/timeline`
- `POST /operator/api/customers/sync`

响应包含稳定 `customer_id`、Base 客户记录链接（已有映射时）、妙搭客户详情链接、同步状态与下一步。写接口必须要求幂等键。公网网关只开放这些最小 Operator 路径，不开放管理员、飞书回调或原始数据库接口。

### 7. 真实 Base 非破坏迁移

目标 Base：`RVtDb7nGkabAMbsDkA0cvxdOnld`。必须先解析并读取真实 block/table/field/view 结构与计数，再决定复用或新增。不得依据文档中的旧名称猜 ID。

优先复用现有客户级 `客户跟进表`；只有真实结构证明安全时才重命名为 `客户 CRM`。新增 `跟进记录` 表与必要视图。任何 schema 写入前保存结构证据；任何业务记录验证前保存原值，完成后恢复测试值。
