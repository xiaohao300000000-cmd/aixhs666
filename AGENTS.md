# Codex 工作规则

## 1. 每次开始前必须阅读

按顺序阅读：

1. `README.md`
2. `docs/PRD.md`
3. `docs/ARCHITECTURE.md`
4. `docs/DATA_MODEL.md`
5. `docs/ORCHESTRATION.md`
6. `TASKS.md`
7. `PROJECT_DASHBOARD.md`
8. `DECISIONS.md`
9. `HANDOFF.md`

不得仅依赖聊天上下文。

## 2. 单任务执行

每次只执行 `TASKS.md` 中一个状态为 `TODO` 的任务。

完成后必须停止，等待下一次指令。

不得擅自连续完成多个任务。

## 3. 开始任务前

必须先输出：

- 当前任务 ID
- 任务目标
- 准备修改的文件
- 可能影响的模块
- 需要运行的测试

## 4. 完成任务后

必须：

1. 运行测试
2. 更新 `TASKS.md`
3. 更新 `HANDOFF.md`
4. 如有新决策，更新 `DECISIONS.md`
5. 报告：
   - 修改了什么
   - 测试结果
   - 未完成事项
   - 已知风险
   - 下一任务建议

## 5. 代码原则

- Python 使用类型标注
- 核心业务逻辑必须有测试
- 数据写入必须幂等
- 平台适配与业务逻辑分离
- 不把平台特定字段泄漏到核心业务层
- 不在 n8n 中实现核心业务逻辑
- 不把密钥写入仓库
- 不静默吞掉异常
- 不擅自加入 Redis、Kafka、Kubernetes
- 不为了“完整架构”提前实现未进入当前阶段的功能

## 6. 文档原则

遇到以下情况必须写入文档：

- 新增依赖
- 修改数据模型
- 修改接口
- 修改任务状态
- 修改采集策略
- 修改默认限制
- 修改项目范围
- 新增平台

重要决定写入 `DECISIONS.md`，当前状态写入 `HANDOFF.md`。

## 7. 平台采集原则

- 只处理正常公开可见的数据
- 不绕过私密权限
- 不实现验证码破解
- 登录失效时暂停并记录
- 页面字段缺失时保存快照并报警
- 平台采集器必须实现统一接口
- 必须支持断点续传和重试

## 8. 禁止事项

- 不得一次重写整个项目
- 不得无测试修改核心模型
- 不得删除已有数据迁移
- 不得在未记录决策时更换技术栈
- 不得把聊天中的临时想法直接当正式需求
- 不得把所有功能塞进一个文件
## 9. 主控与子会话

项目采用主控会话和执行子会话模式。

详细规则见 `docs/ORCHESTRATION.md`。

核心要求：

- 主控会话负责派单、验收、合并和全局进度
- 子会话只执行一个任务
- 子会话不得修改全局进度文件
- 每个子任务必须有任务简报和子任务报告
- 并行任务必须使用独立分支或 worktree
- 只有主控会话可以把任务标记为 DONE

## 10. 项目进度更新

主控会话在任务验收后必须同步更新：

1. `TASKS.md`
2. `PROJECT_DASHBOARD.md`
3. `HANDOFF.md`
4. 必要时更新 `DECISIONS.md`

不得只在聊天中宣布任务完成。

## 独立执行对话

项目经理对话可以长期保留。

每个任务的执行对话必须：

- 单独新开
- 不从项目经理对话派生
- 不继承项目经理聊天上下文
- 只接收当前任务包和点名文件

仓库文件负责协作，聊天上下文不负责协作。
