# 多子会话并发管理策略

## 1. 目标

允许项目经理主控会话同时管理多个执行子会话，提高开发速度，同时避免：

- 多个子会话修改同一文件
- 数据模型和接口相互冲突
- 主控来不及验收
- 子任务完成后无人合并
- 项目状态失真
- 聊天很多，代码没前进

并发数量不固定，由主控会话根据任务独立性、当前阻塞和自身管理能力动态决定。

## 2. 默认并发规则

新电脑首次运行时：

```text
默认并发：2 个执行子会话
建议上限：3 个
硬上限：4 个
```

只有满足以下条件，才从 2 个提高到 3–4 个：

- 最近两个子任务均按时提交完整报告
- 主控能在一个工作周期内完成验收
- 没有积压超过 2 个待验收任务
- 没有文件冲突
- 测试时间和电脑资源仍然稳定
- 项目看板与实际分支状态一致

如果出现以下任一情况，应降低并发：

- 待验收任务达到 3 个
- 两个子任务修改相同公共接口
- 合并后连续出现回归错误
- 主控无法及时更新项目文档
- 电脑内存持续紧张
- Codex 会话频繁丢失任务边界
- 某个核心架构决策尚未完成

## 3. 主控管理容量

主控的真实容量不是“能开多少窗口”，而是：

```text
管理容量 =
可及时验收的任务数
× 可避免冲突的任务独立度
× 文档更新及时性
```

如果主控只能同时验收 2 个任务，那么开 6 个子会话不会加速，只会制造 6 份待办遗产。

## 4. 可并行任务判断

同时满足以下条件才允许并行：

1. 所有前置依赖已完成
2. 不修改相同文件或目录
3. 不修改同一数据库迁移
4. 不修改同一公共接口
5. 不同时改变同一配置文件
6. 每个任务都有独立测试
7. 每个任务有独立分支或 worktree
8. 主控能在任务结束后及时验收

## 5. 推荐并发组合

### 阶段 0

可并行：

```text
T02 核心数据模型
T04 PlatformAdapter 与 Mock
```

条件：T01 完成。

之后：

```text
T03 任务状态机
T05 去重与发现关系
```

不能完全无脑并行，主控需要明确文件边界。

### 阶段 1

大部分任务依赖较强：

```text
T07 → T08 → T09 → T10
```

不建议强行并行。

可额外并行：

- 采集器测试快照
- 文档更新
- 页面结构研究
- 监控与日志模块

### 阶段 2–4

适合并行的组合示例：

```text
T12 评论区动态预算
T14 文本处理与低信息标记
```

```text
T17 事件日历
T18 需求事件链
```

前提是依赖已满足且不修改同一模型文件。

## 6. 主控派发算法

主控每轮按以下顺序操作：

1. 读取 `TASKS.md`
2. 筛选依赖已完成的 TODO 任务
3. 检查 `orchestration/FILE_LOCKS.md`
4. 检查当前活跃 Worker 数量
5. 检查待验收报告数量
6. 计算还能派发多少任务
7. 优先选择互不冲突的任务
8. 创建任务简报
9. 分配独立 worktree
10. 更新 Worker 注册表和进度看板

## 7. 并发槽位

主控维护四个并发槽位：

| 槽位 | 默认状态 | 用途 |
|---|---|---|
| W1 | 可用 | 核心开发任务 |
| W2 | 可用 | 独立开发或测试任务 |
| W3 | 关闭 | 稳定后启用 |
| W4 | 关闭 | 仅低冲突阶段启用 |

主控不得因为槽位存在就强行填满。

## 8. 待验收队列

建议限制：

```text
待验收报告上限：2
```

达到上限后：

- 暂停派发新任务
- 优先完成代码审查、测试和合并
- 更新项目看板
- 处理冲突与返工

这条很重要。项目经理最容易犯的病，就是热衷派活，不热衷验收，最后仓库里堆着一排“差不多完成”的东西。

## 9. 文件锁

主控使用 `orchestration/FILE_LOCKS.md` 登记：

- 被哪个任务占用
- 锁定的文件或目录
- 开始时间
- 预计释放条件
- 是否允许共享只读

子会话开始前必须检查文件锁。

## 10. 分支与 Worktree

每个子会话使用独立分支：

```text
task/TXX-short-name
```

推荐使用 Git worktree：

```bash
git worktree add ../worktrees/TXX -b task/TXX-short-name
```

主控仓库保留在主分支，用于：

- 查看全局状态
- 生成任务简报
- 验收
- 合并
- 更新文档

子会话不得直接操作主控工作区。

## 11. 并发效果评估

每完成一轮并发任务，主控记录：

- 并发数量
- 完成任务数
- 验收耗时
- 冲突次数
- 回归错误数
- 返工次数
- 主控积压数量

如果并发增加后：

- 总完成速度没有提高
- 冲突和返工明显增加
- 主控验收持续积压

则降低并发。

## 12. 推荐起步方式

在新电脑上：

### 第一轮

```text
并发：1
任务：T01
```

先验证环境、Docker、Git、Codex 和测试流程。

### 第二轮

T01 完成后：

```text
并发：2
Worker 1：T02
Worker 2：T04
```

### 第三轮

如果第二轮没有冲突：

```text
并发：2–3
```

不要第一天就开 4 个。新电脑不是请来了四名靠谱员工，只是更有能力同时展示四个错误窗口。
