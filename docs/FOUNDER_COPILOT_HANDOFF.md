# Founder Copilot 专用 Handoff

更新时间：2026-07-15

## 1. 给下一个 Codex 的强制说明

用户明确要求：后续 Codex 在完成项目任务的同时，观察用户在表达完整性、产品建立、决策推进和协作方式上的可改进点，并约每 2–3 天提供一次简短反馈。具体时机由 Codex 根据证据判断，不需要机械按日历发送。

开始任何相关工作前，必须阅读：

1. `docs/FOUNDER_COPILOT.md`
2. 本文档
3. `HANDOFF.md`
4. `TASKS.md`

## 2. 当前状态

- Founder Copilot 目前只有设计和协作规则，尚未实现自动化数据模型、定时任务或飞书反馈卡。
- 飞书 AI 客户线索表已经有待确认、已确认、高意向、已忽略和画册视图。
- 当前 Base 只有一个停用的空工作流，人工审核工作台尚未实施。
- 最近一次正式成长观察基线：`2026-07-15`，记录在 `docs/FOUNDER_COPILOT.md` 第 10 节。

## 3. 每次任务中的观察方式

Codex 应静默记录以下证据，但不要因此降低主任务执行质量：

- 用户如何描述目标、用户、场景、约束和成功标准。
- 用户是否发现技术成功与产品成功之间的差异。
- 用户如何处理失败、证据不足、重复尝试和方向切换。
- 用户是否主动补充结果消费、人工操作和后续流转。
- 用户当前更需要结论、解释、方案收敛还是执行。

只记录实际发生的事实。不要推断心理健康、人格、智力或其他敏感属性。

## 4. 何时发送反馈

推荐条件：

- 距离上次反馈约 2–3 天；并且
- 至少完成了两个有实质内容的任务，或出现一个重复两次以上的高价值模式。

可以由 Codex提前反馈：

- 当前模式正在造成明显返工。
- 用户主动询问自身表现或改进方向。
- 一个新的产品思维突破值得及时强化。

应当暂缓：

- 正在处理线上故障、安全问题或用户明显只需要快速执行。
- 没有新证据。
- 反馈只能写成空泛鼓励。

## 5. 反馈输出合同

使用以下格式，控制在 4 个短段落以内：

```text
成长观察

做得好的：<具体事实>
最值得提升：<一个问题>
为什么重要：<对产品或执行的影响>
下次可以这样表达：<可直接使用的示例>
```

规则：

- 一次只反馈一个主要改进点。
- 必须有最近任务中的具体证据。
- 不进行居高临下的评价。
- 不把用户正常的着急或不满描述成性格缺陷。
- 不为了鼓励而夸大，也不为了显得深刻而挑错。

## 6. 如何持久化下一次观察

当发送一次正式成长反馈后：

1. 在 `docs/FOUNDER_COPILOT.md` 的“观察记录”区域追加日期、证据、优势、改进点和表达示例。
2. 更新本文档的“最近一次反馈”日期。
3. 如反馈形成新的长期规则，更新 `DECISIONS.md`。
4. 将相关文档提交并推送 GitHub，确保下一个 Codex 能继续。

最近一次反馈：`2026-07-15` 初始基线。

## 7. 人工审核工作台后续任务建议

建议拆分为独立 V17 任务：

1. `V17-01`：补齐客户线索主表审核字段和卡片视图。
2. `V17-02`：新增审核记录表，建立与客户线索的关联。
3. `V17-03`：创建确认有效、确认无效、待二审和重新分析工作流。
4. `V17-04`：将关键审核结果幂等同步回 PostgreSQL。
5. `V17-05`：统计 AI/人工一致率、误判原因和 Campaign 审核效果。

工作流不得在飞书内部直接运行完整 DeepSeek 流程。重新分析必须只创建持久任务，由独立 Worker 执行。

## 8. 禁止事项

- 不把 Founder Copilot 变成每轮都出现的固定说教模块。
- 不在证据不足时生成评价。
- 不记录或推断用户的敏感心理、健康或人格信息。
- 不让成长反馈盖过当前任务结果。
- 不把 Base 当作唯一事实源；业务状态仍以 PostgreSQL 为准。
