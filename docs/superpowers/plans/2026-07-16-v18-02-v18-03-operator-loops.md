# V18-02/V18-03 Operator Loops Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the Miaoda lead-review and task-center placeholders with real PostgreSQL-backed operator workflows that remain connected to the existing workbench and Skill Runtime.

**Architecture:** Extend the isolated FastAPI Operator API with narrowly scoped lead-review and Skill Run resources, keep authorization at the gateway boundary, and proxy every browser request through the existing NestJS same-origin BFF. The React console will use route-level pages but share typed API models, refresh workbench counts after writes, and never duplicate business facts into the Miaoda database.

**Tech Stack:** Python 3.12, FastAPI, SQLAlchemy 2, PostgreSQL/SQLite tests, React 19, TypeScript 5.9, NestJS 10, Jest, Tailwind CSS, Miaoda full-stack hosting, Tailscale Funnel.

## Global Constraints

- PostgreSQL remains the only fact source for leads, human review results, Skill Runs, events, and results.
- The public gateway exposes only `/health` and `/operator/api/*`; existing `/api/leads`, admin APIs, and Feishu callbacks remain unreachable.
- Browser code never receives `OPERATOR_API_TOKEN`; NestJS injects it server-side.
- Lead writes are idempotent for repeated identical actions and update both `Lead` and the newest matching `LeadScreeningResult` where available.
- Task execution exposes only registered safe Skill definitions; the first usable template is `screen_historical_leads`.
- No collection, comment, private-message, or other external-platform write is triggered by lead review.
- All write operations return the updated resource so the UI can advance without a full reload.
