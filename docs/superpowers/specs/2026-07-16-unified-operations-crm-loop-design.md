# 妙搭 × 飞书 × Base CRM 统一运营闭环设计

日期：2026-07-16  
状态：用户已确认，等待书面规格复核  
目标：把现有妙搭工作台、任务中心、线索审核、飞书多维表格、飞书消息与小红书公开回复连接成一个真实可用的个人运营闭环。

## 1. 设计结论

系统采用以下职责划分：

- 妙搭是主操作台，承载今日行动、任务结果、连续审核、客户详情、话术确认、真实发送和排障入口。
- PostgreSQL 与业务后端是唯一事实源，所有入口调用同一套业务命令。
- 飞书多维表格是个人 CRM 协作层，承载客户列表、跟进记录、筛选视图和轻量人工维护。
- 飞书消息负责提醒、快捷动作和深链，不维护独立业务状态。
- Windows 小红书浏览器负责真实采集、公开评论回复和回复检查。
- 第一阶段只打通小红书公开评论回复；小红书私信保留统一渠道接口，不在本阶段承诺真实发送。

系统必须围绕同一条业务流水线工作：

```text
Campaign
→ 客户发现任务
→ 运行结果报告
→ 候选线索池
→ 每日审核队列
→ 推进为客户
→ Base CRM
→ 公开回复草稿
→ 人工确认
→ 真实发送
→ 等待客户回复
→ 定时检查新回复
→ 再次人工确认与回复
→ 业务复盘
```

任何页面和入口都必须回答四件事：

1. 发生了什么；
2. 为什么发生；
3. 结果去了哪里；
4. 下一步应该做什么。

## 2. 用户与范围

### 2.1 用户

第一阶段只有一名运营者使用，默认运营者即当前登录用户。

不实现：

- 多人客户分配；
- 角色权限；
- 团队协作规则；
- 业绩排名；
- 销售额统计；
- 部门或销售人员看板。

所有数据结构仍保留未来增加负责人和多人协作的扩展空间，但首期界面不重复展示默认负责人。

### 2.2 核心成功标准

系统上线后，用户应能完成以下最小真实闭环：

1. 自动或手动运行一次客户发现任务；
2. 查看人类可理解的结果报告；
3. 从结果进入本次产生的审核队列；
4. 将真实候选推进为客户；
5. 在 PostgreSQL、妙搭和 Base CRM 中看到同一客户；
6. 自动生成公开回复草稿；
7. 人工确认后点击发送；
8. Windows 浏览器完成真实公开回复；
9. 系统从平台回读并确认发送结果；
10. 妙搭、Base CRM 和飞书卡片同步显示发送结果；
11. 定时检查目标评论是否出现客户新回复；
12. 发现新回复时创建下一步行动并立即飞书提醒。

不能以写入测试数据库、模拟浏览器返回、前端假成功或仅更新卡片替代真实闭环验收。

## 3. 产品原则

### 3.1 行动优先

妙搭首先是行动型运营驾驶舱，不是工程监控台。页面默认展示业务行动与业务结果，技术状态下沉到“系统健康”和折叠的“技术详情”。

### 3.2 高召回优先

筛选目标是尽量不遗漏潜在客户。低置信度、地区未知、信息不足、只有单条证据等情况不得直接作为自动排除理由。

系统优先通过排序控制审核顺序，而不是通过删除控制数量。

### 3.3 后果明确

用户点击任何业务动作前，页面必须说明将产生的后果。动作完成后，页面必须返回已创建或更新的客户、跟进事项、Base 记录、飞书提醒和下一步入口。

### 3.4 人工确认外发

系统可以自动生成草稿，但不得自动向客户发送任何公开回复或私信。外发必须经过明确人工确认，并由用户点击最终发送按钮。

### 3.5 一个事实源，多种入口

妙搭、飞书卡片和 Base CRM 可以提供不同交互入口，但不能各自实现不同的业务后果。所有动作必须进入统一业务命令层。

### 3.6 失败可恢复

业务主动作与外围同步解耦。客户已创建但 Base 同步失败时，客户创建保持成功，Base 同步进入重试；不得因为外围失败回滚已确认的业务事实。

## 4. 当前能力与真实可行性

### 4.1 已具备、可直接复用

- PostgreSQL 数据模型、任务表、Worker 心跳和持久任务机制；
- `skill_runs` / `skill_run_events` 与 Skill Run Worker；
- 妙搭全栈应用、NestJS BFF、FastAPI Operator API 和受保护公网网关；
- 真实线索队列、证据展示和人工审核写回；
- AI 筛选结果到 Base 候选线索与证据表的幂等同步；
- 客户跟进 Base 同步服务；
- 公开评论回复生成、飞书审批卡、持久发送任务与 Windows 远程浏览器发送器；
- 评论发送的 `sent`、`failed`、`result_unknown` 保护和人工恢复流程；
- 飞书消息卡片和回调基础设施；
- Tailscale Funnel、LaunchAgent 与本地 Worker 常驻运行能力。

### 4.2 已有但必须统一

- 妙搭审核路径目前主要修改 Lead 与 Screening 状态；飞书回调路径会额外创建外联或评论回复，两个入口后果不一致；
- Base 候选线索同步与客户跟进同步已存在，但没有被同一个客户推进命令编排；
- 任务结果已记录真实计数和事件，但妙搭默认展示原始 JSON；
- 公开回复已经有真实执行链路，但确认、发送、客户阶段、Base CRM 和妙搭客户中心尚未形成同一条产品流程；
- 系统已有 Lead、Screening、Evidence、Comment Reply 和 Task 等事实，但缺少统一客户时间线视图和明确的下一步行动模型。

### 4.3 尚未具备

- 统一业务命令与原子编排；
- 任务结果的人类可读报告与候选明细钻取；
- 高召回分层队列、每日 50 条审核计划和排除抽检；
- 客户详情与统一时间线；
- Base CRM 允许字段的可靠双向同步；
- 每日 14:00 自动发现任务；
- 每日 15:00、21:00 的公开回复检查；
- 发现客户新回复后的入库、提醒和下一次草稿生成；
- 私信渠道适配器的真实实现。

### 4.4 真实可行性边界

公开回复真实发送依赖：

- Windows 浏览器在线；
- 小红书账号登录有效；
- 远程 CDP 可连接；
- 当前页面 selector 可识别；
- 目标帖子或评论仍存在；
- 平台未限制当前账号操作。

因此产品必须把“发送能力是否可用”作为运行时事实展示，不能仅根据配置存在就显示为可用。上线前必须完成真实 selector probe 和一条经用户明确批准的真实公开回复验收。

回复监测能否完整验收取决于是否存在可控的新回复。若没有真实客户及时回复，必须使用预先批准的受控测试账号在目标评论下产生真实回复；不能用数据库插入或 Mock 事件宣称验收通过。

## 5. 统一业务对象

### 5.1 Campaign

描述一次持续获客目标，包括地区、课程或考试、人群、需求类型、时间范围、来源范围和筛选策略。

首期默认 Campaign：

```text
名称：福州 KET/PET 潜在客户发现
地区：福州
课程/考试：KET、PET
人群：家长或明确需求者
需求：机构比较、报名、课程咨询、价格、考试准备
时间范围：最近 30 天
来源：公开帖子、评论和回复
筛选策略：高召回优先
```

### 5.2 Run

一次 Campaign 执行批次。Run 保存参数快照、输入范围、进度、候选结果、排除结果、同步结果和业务报告。

### 5.3 Candidate Lead

审核前候选。候选关联一个公开平台用户、一个或多个证据、产生它的 Run 和 Campaign。

### 5.4 Customer

人工推进后的正式客户。第一阶段可在现有 `Lead` 事实基础上扩展客户阶段与时间线，不要求为了命名纯粹而立即复制一张 Customer 表；是否新增独立 Customer 表应在实施计划中根据现有外键和迁移成本决定。

无论物理表如何实现，API 和产品层统一使用 `customer_id`，不得把 Base record ID、Screening ID 或平台用户 ID 当客户主键。

### 5.5 Follow-up Action

表示用户需要完成的一次下一步行动，例如首次联系、处理客户新回复、三天未回复复查或确认结果不确定的发送。

### 5.6 Contact Attempt

一次经过确认的联系尝试，保存渠道、目标、内容版本、确认记录、发送任务、发送结果和证据。

### 5.7 Timeline Event

面向用户的客户时间线事件。事件由后端事实生成，不允许前端拼接一套不可审计的临时时间线。

## 6. 统一状态机

### 6.1 候选状态

```text
priority_review
standard_review
uncertain_review
deferred
rejected
promoted
```

- `priority_review`：高优先级待审核；
- `standard_review`：普通待审核；
- `uncertain_review`：不确定但保留审核；
- `deferred`：人工暂缓，必须带重新出现条件；
- `rejected`：人工或明确规则淘汰；
- `promoted`：已推进到客户。

### 6.2 客户 CRM 阶段

```text
new_customer
awaiting_first_contact
contact_approved
awaiting_send
contact_sent_waiting_reply
customer_replied
in_conversation
high_intent
converted
deferred
temporarily_unreachable
invalid
```

中文展示：

```text
新客户
待首次联系
话术已确认
等待发送
已联系待回复
客户已回复
沟通中
有明确意向
已转化
暂缓
暂时失联
无效
```

### 6.3 联系尝试状态

```text
draft
awaiting_approval
approved
queued
sending
sent
failed
result_unknown
cancelled
```

编辑已确认内容后，状态必须回到 `awaiting_approval`，原确认版本保留审计但不可发送。

### 6.4 回复检查状态

```text
scheduled
checking
no_new_reply
new_reply_found
login_required
target_missing
result_unknown
failed
```

## 7. 统一业务命令

以下动作必须由后端命令服务实现，妙搭、飞书和 Base 不得自行拼接副作用：

### 7.1 `promote_candidate_to_customer`

输入：候选 ID、审核理由、操作人、幂等键。  
后果：

1. 写入人工审核结论；
2. 创建或合并客户；
3. 关联候选和证据；
4. 设置客户阶段为“待首次联系”；
5. 创建首次跟进事项；
6. 生成首次公开回复草稿；
7. 请求 Base CRM upsert；
8. 更新原飞书卡片；
9. 写入客户时间线；
10. 返回客户、跟进事项、草稿和 Base 同步状态。

### 7.2 `defer_candidate`

必须保存原因以及重新出现条件：指定日期、新证据出现或人工恢复。没有重新出现条件时不得完成暂缓。

### 7.3 `reject_candidate`

必须保存结构化原因。淘汰的是当前候选判断，不永久封禁平台用户；新证据允许重新激活。

### 7.4 `approve_contact_draft`

冻结内容版本、目标、渠道、确认人和确认时间。确认后仍不发送。

### 7.5 `send_approved_contact`

验证确认版本仍有效，创建唯一持久发送任务，并返回排队状态。重复点击使用同一幂等键，不得重复发送。

### 7.6 `record_contact_result`

由真实发送 Worker 调用，写入 `sent`、`failed` 或 `result_unknown`，更新客户阶段、跟进记录、Base CRM、飞书卡片和下一步事项。

### 7.7 `record_customer_reply`

保存平台新回复、关联原始联系尝试、更新客户阶段、创建处理事项、生成下一次草稿并发送飞书提醒。

## 8. 阶段 1：今日工作台

### 8.1 页面目标

用户打开后 10 秒内知道：

- 今天最重要的事项；
- 今日 50 条审核进度；
- 是否有客户新回复；
- 哪些客户需要联系；
- 是否存在阻塞真实业务的异常；
- 点击哪里可以继续。

### 8.2 今日行动队列

默认排序：

```text
人工确认后的真实发送
> 客户新回复处理
> 到期客户跟进
> 高优先级线索审核
> 普通线索审核
> 不确定项与排除抽检
> 非阻塞系统异常
```

如果系统异常会阻止发送或回复检查，该异常提升到对应业务动作之前。

每日行动队列基本稳定，只允许客户新回复、到期跟进、已确认发送和阻塞异常插队。普通分数变化不在用户操作过程中不断重排队列。

### 8.3 工作台卡片

- 今日最重要的事；
- 今日审核 `已完成/50`；
- 客户有新回复；
- 今天待首次联系；
- 已联系待回复；
- 超过三天未回复；
- 最近任务结果；
- 15:00、21:00 回复检查状态；
- 今日业务成果；
- Base CRM 快捷入口；
- 系统健康摘要。

Worker、原始任务类型、API 状态和 JSON 不在首页默认展开。

## 9. 阶段 2：任务中心

### 9.1 业务入口

模板使用业务语言：

- 寻找新的高意向客户；
- 扩大潜在线索池；
- 重新分析不确定线索；
- 检查已有客户的新需求信号。

第一阶段默认“扩大潜在线索池”。

### 9.2 运行前预览

预览必须显示：

- Campaign 条件；
- 预计分析数量；
- 新增与重复数据范围；
- 预计候选区间；
- 每日审核队列上限；
- 是否访问平台；
- 是否会对外写入；
- 明确声明不会自动联系客户。

技术参数收进“高级设置”。

### 9.3 自动运行

默认每天北京时间 14:00 自动运行“福州 KET/PET 潜在客户发现”，同时保留“立即运行”。自动运行只处理新增或变化数据。

## 10. 阶段 3：运行结果报告

### 10.1 一句话结论

结果页顶部使用业务结论，例如：

> 本次分析 300 条公开内容，发现 72 条候选线索，其中 18 条为高优先级，今日审核队列已准备 50 条。

没有候选时显示：

> 本次运行已完成，但没有发现符合当前 Campaign 的客户。

`succeeded` 只作为技术详情，不作为主要结论。

### 10.2 业务漏斗

```text
分析内容
→ 发现可能需求
→ 高优先级候选
→ 普通候选
→ 不确定候选
→ 自动排除
→ 进入今日审核
→ 已同步 Base
```

每个数字可钻取到明细。

### 10.3 明细与去向

结果页展示：

- 候选客户；
- 推荐原因和证据；
- 排除原因统计；
- 需要时展开的排除明细；
- PostgreSQL、妙搭审核队列、Base 和飞书提醒的去向；
- 下一步主按钮“审核本次候选”。

原始 JSON、事件和模型返回移入默认折叠的“技术详情”。

## 11. 高召回筛选与每日 50 条

### 11.1 分层

- 高优先级审核；
- 普通审核；
- 不确定审核；
- 明确自动排除。

### 11.2 自动排除限制

仅允许高确定性排除：明确广告或机构账号、明确无关、完全重复、已删除且无有效证据、明确违反硬条件、测试数据、人工确认无效且没有新证据。

以下不得自动排除：信息不足、地区未知、意向弱、低置信度、单条证据、课程不明确。

### 11.3 每日审核预算

每日默认 50 条，使用以下确定性填充规则：

```text
质量控制保留位          5 条
按业务优先级填充位      45 条
```

- 质量控制保留位优先放入不确定候选；不确定候选不足时放入自动排除抽检；两者均不足时回填普通候选。
- 其余 45 条按高优先级、普通、不确定的业务排序填充，不设置“高优先级最多 30 条”的硬上限。
- 当出现必须立即处理的强信号时，允许紧急候选插队并使当日队列暂时超过 50 条，但必须明确标记为紧急新增。

每日 50 条是展示预算，不是保存上限。第 51 条以后继续保存，并在后续日期重新排序。完成 50 条后提供“继续审核 20 条”和“只看高优先级”。

### 11.4 防误杀

- 定期抽检自动排除项；
- 保存 AI 原判断和人工纠正；
- 统计被恢复的线索和误排规则；
- 新证据允许历史客户重新激活；
- 同一客户多条证据合并为一个审核位置。

## 12. 阶段 4：线索审核

### 12.1 页面布局

- 左侧：稳定排序的审核队列；
- 中间：原文、评论、用户资料、AI 摘要、匹配规则和风险；
- 右侧：动作后果预览；
- 底部固定动作：推进为客户、暂缓判断、淘汰线索。

### 12.2 推进为客户

“有效”和“进入跟进”合并为“推进为客户”。完成后必须显示：

- 客户编号；
- CRM 阶段；
- 首次跟进事项；
- 草稿状态；
- Base CRM 同步状态；
- 打开客户详情；
- 打开 Base 记录；
- 继续审核下一条。

### 12.3 暂缓与淘汰

暂缓必须选择原因和重新出现条件。淘汰必须选择结构化原因，并保留恢复入口和审计记录。

## 13. 阶段 5：客户中心

### 13.1 客户详情

顶部显示客户阶段、下一步、下次检查、最近联系和关键需求。页面包含：

- 客户摘要；
- 当前行动；
- 完整证据；
- 联系草稿；
- 发送状态；
- 客户时间线；
- Base CRM 链接；

### 13.2 客户时间线

### 13.3 重复客户

平台用户唯一键相同时优先合并；无法确定时进入“可能重复”人工确认。不得因为同一客户出现新内容而创建重复 CRM 客户。

## 14. Base CRM

### 14.1 定位

Base CRM 就是现有飞书多维表格中的个人 CRM，不是新安装的软件。PostgreSQL 仍是唯一事实源。

### 14.2 表结构

保留并明确：

- `AI筛选客户线索`：审核前候选池；
- `AI筛选证据明细`：原始证据与判断依据；
- `系统控制台`：系统状态与同步异常。

调整或新增：

- 将现有客户级 `客户跟进表` 明确为 `客户 CRM`，一位客户一行；
- 新增 `跟进记录`，一次联系、回复或人工操作一行；
- 可选新增 `运行批次`，保存人类可读的 Run 摘要和妙搭结果链接。

任何字段调整必须先读取真实表结构与现有记录，采用非破坏迁移，不直接删除旧字段或旧记录。

### 14.3 客户 CRM 字段

- 妙搭详情链接；
- 同步版本；
- 同步状态；
- 最近同步时间。

### 14.4 跟进记录字段

- 跟进记录 ID；
- 关联客户；

- 操作类型；
- 联系渠道；

### 14.5 双向同步

Base 人工可修改：

- CRM 阶段；
- 下次跟进时间；
- 跟进备注；
- 联系结果；
- 客户标签。

系统只读：

- 后端客户 ID；
- AI 判断；
- 原始证据；
- 来源链接；
- Campaign；
- 运行批次；
- 发送状态和证据；
- 同步版本。

Base 修改通过允许字段白名单写回后端。首期由后端同步 Worker 每 5 分钟增量读取一次允许人工维护的字段，同时提供妙搭“立即同步”入口；系统事实字段始终以后端为准。同步失败只重试同步，不重放业务动作或发送任务。

## 15. 公开回复与渠道接口

### 15.1 两步确认

第一步“确认此话术”冻结内容版本、回复目标、渠道和确认记录。  
第二步“发送公开回复”再次展示目标和内容，用户确认后创建持久发送任务。

任何编辑都会使原确认失效。

### 15.2 统一渠道协议

业务层使用统一渠道能力：

```text
validate_target
check_availability
prepare_send
send
verify_result
classify_failure
safe_to_retry
```

渠道标识预留：

```text
xiaohongshu_public_reply
xiaohongshu_direct_message
manual_phone
manual_wechat
email
```

第一阶段仅实现并验收 `xiaohongshu_public_reply`。私信接口保留，但 UI 不得显示为已连接或可发送。

### 15.3 发送结果

发送成功后自动：

- 客户阶段改为“已联系待回复”；
- Base CRM 更新最近联系时间；
- Base 跟进记录新增公开回复；
- 妙搭客户时间线新增发送事件；
- 飞书卡片更新；
- 工作台待联系减少、等待回复增加；
- 保存目标链接、时间、执行账号和平台证据；
- 创建后续回复检查事项。

`result_unknown` 不得自动重试。必须先平台核对或由人工确认未发送。

## 16. 回复检查

### 16.1 定时计划

北京时间 `Asia/Shanghai`：

- 15:00 第一次检查；
- 21:00 第二次检查。

只检查“已联系待回复”、目标有效且尚未发现新回复的客户。

### 16.2 新回复处理

发现新回复后：

- 保存回复原文、时间、来源链接和唯一标识；
- 客户阶段改为“客户已回复”；
- Base 跟进记录新增“收到客户回复”；
- 工作台新增“客户回复待处理”；
- 生成下一次回复草稿；
- 立即发送飞书提醒；
- 21:00 的提醒提供“现在处理”和“明天提醒我”。

下一次回复仍必须人工确认和点击发送。

### 16.3 无回复与超期

- 没有新回复只更新最近检查与下次检查，不发送飞书消息；
- 3 天未回复进入超期视图；
- 7 天建议再次回复、暂缓或结束；
- 14 天默认建议“暂时失联”，不自动删除客户。

### 16.4 幂等

同一平台回复不得重复入库、重复提醒、重复创建跟进事项或重复生成相同草稿。

## 17. 调度与资源互斥

北京时间：

```text
14:00 客户发现任务
15:00 第一次回复检查
21:00 第二次回复检查
```

同一个 Windows 浏览器同一时间只执行一个平台任务。

任务优先级：

```text
人工确认的真实发送
> 客户回复检查
> 客户发现任务
```

如果 14:00 的发现任务在 15:00 尚未完成：

1. 在安全断点暂停；
2. 执行回复检查；
3. 检查完成后从断点继续；
4. 已完成数据不得重复处理。

页面必须解释暂停原因，不得把正常让行显示为失败。

## 18. 阶段 6：业务复盘

不做业绩统计，展示：

- 分析公开内容数量；
- 候选线索数量；
- 人工审核数量；
- 推进客户数量；
- 完成公开回复数量；
- 客户新回复数量；
- 继续沟通数量；
- 高、普通、不确定候选推进率；
- 排除抽检错误率；
- 被人工恢复数量；
- Campaign 和来源的候选、客户、回复漏斗。

系统可以生成每周建议，但不得未经人工确认自动修改筛选规则。

## 19. API 与页面边界

### 19.1 Operator API

实施时应新增或重构为资源与命令分离的接口，至少覆盖：

- 今日行动与进度；
- Run 业务报告、候选明细和排除明细；
- 按 Run 创建审核会话；
- 推进、暂缓、淘汰候选；
- 客户详情与时间线；
- 联系草稿生成、编辑、确认、发送；
- 发送任务状态；
- 回复检查状态与立即检查；
- Base 同步状态与重试；
- 系统能力可用性。

具体 URL 在实施计划中遵循现有 Operator API 风格确定，但所有写操作必须使用幂等键和明确命令语义。

### 19.2 NestJS BFF

BFF 继续隐藏 Operator token，负责同源认证、参数校验、错误翻译和安全超时，不复制 Python 业务编排。

### 19.3 妙搭页面

- `/`：今日工作台；
- `/tasks`：业务任务与结果；
- `/leads`：连续审核；
- `/customers`：客户 CRM 列表；
- `/customers/:id`：客户详情和时间线；
- `/system-health`：技术连接、Worker、同步和恢复。

飞书和 Base 深链必须落到具体 Run、候选、客户或行动，不只打开首页。

## 20. 交互体验要求

### 20.1 用户语言

默认页面不得出现需要用户理解的原始 JSON、内部 task type、数据库状态名或异常堆栈。技术内容只在折叠详情中提供。

### 20.2 操作连续性

- 审核后自动进入下一条；
- 推进后可直接打开客户或继续审核；
- Run 结果一键进入本次候选；
- Base 和飞书一键进入对应妙搭详情；
- 页面刷新后保持真实任务状态；
- 长任务离开页面后继续运行；
- 返回页面后恢复到原批次与筛选条件。

### 20.3 反馈

禁止只显示“操作成功”。必须展示具体后果，例如：

> 已创建客户 #151，首次跟进已生成，Base CRM 已同步，公开回复草稿等待确认。

部分成功必须逐项说明：

> 客户已创建，草稿已生成；Base 同步失败，后台正在重试。

### 20.4 防误操作

- 外发采用两步确认；
- 最终发送按钮显示渠道、目标和内容；
- 重复点击不重复发送；
- 暂缓需要原因和恢复条件；
- 淘汰需要原因并可恢复；
- `result_unknown` 必须人工核对；
- Base 系统事实字段不可覆盖后端事实。

## 21. 真实验收标准

### 21.1 自动化验证

- 统一命令单元测试；
- 幂等与重复点击测试；
- 状态机合法与非法迁移测试；
- Base 双向同步冲突测试；
- 发送 `sent`、`failed`、`result_unknown` 测试；
- 回复检查去重测试；
- 调度优先级和断点恢复测试；
- Operator API 与 BFF 契约测试；
- 妙搭关键交互测试；
- 全量现有测试回归。

### 21.2 最小真实闭环

必须用真实 PostgreSQL、真实 Base、真实飞书、真实妙搭发布环境、真实 Windows 浏览器和经用户批准的真实小红书目标完成：

1. 自动或手动创建 Run；
2. 产生真实候选；
3. 人工推进为客户；
4. Base CRM 创建或更新真实记录；
5. 妙搭生成并编辑真实草稿；
6. 用户确认并点击发送；
7. Windows 浏览器真实公开回复；
8. 平台回读确认回复存在；
9. 三端显示一致状态；
10. 受控账号产生真实新回复；
11. 定时或手动检查发现该回复；
12. 飞书真实提醒；
13. 妙搭和 Base 显示客户新回复与下一步。

任何依赖不可用时必须报告具体阻塞，不得降级为 Mock 后继续宣称真实验收通过。

### 21.3 真人体验验收

验收者以真实运营者身份完成完整流程，并记录：

- 打开工作台 10 秒内能否说出首要任务；
- 是否无需理解 JSON 或内部状态；
- Run 完成后能否找到具体候选和数据去向；
- 审核后能否明确知道客户在哪里；
- 是否能在不跨系统搜索的情况下完成话术确认与发送；
- Base、妙搭、飞书是否显示同一客户与状态；
- 刷新、离开、重开页面后任务是否继续且状态一致；
- 失败提示是否告诉用户下一步；
- 50 条连续审核是否出现明显重复、迷路或操作疲劳；
- 15:00、21:00 检查结果是否准确且不过度打扰。

验收报告必须区分：自动化通过、真实外部链路通过、真人体验通过。

## 22. 实施分解

本设计作为一个产品闭环实施，但按以下可独立验收的顺序交付：

1. **统一命令与客户事实**：统一审核后果、客户时间线、状态机和幂等；
2. **Base CRM**：客户主表、跟进记录、双向同步和深链；
3. **人类结果与高召回队列**：Run 报告、每日 50 条和排除抽检；
4. **妙搭行动体验**：工作台、任务中心、审核页和客户中心；
5. **公开回复闭环**：草稿、两步确认、真实发送和三端状态；
6. **回复检查与调度**：14:00、15:00、21:00 调度、互斥、提醒与回复处理；
7. **真实全流程验收**：受控真实目标、真实 Base/飞书/妙搭/浏览器和真人体验报告。

前一阶段没有通过自动化与最小真实验收时，不进入依赖它的后一阶段。

## 23. 明确不做

- 自动发送未经人工确认的内容；
- 首期真实小红书私信；
- 团队角色与销售业绩；
- 以 Base Workflow 承担核心业务状态机；
- 让妙搭直接连接数据库或保存密钥；
- 用前端状态代替后端事实；
- 为了页面好看隐藏失败或结果不确定；
- 删除原始 JSON、审计事件或历史记录；
- 在真实链路未通过时用模拟结果宣布完成。
