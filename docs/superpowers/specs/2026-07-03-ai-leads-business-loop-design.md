# AI 自动获客最小闭环设计

## 背景

当前系统已经能采集小红书帖子、评论、公开用户和查询来源，并能做文本处理、需求事件链、聚类、查询评分和内容洞察。但产品方向需要从“展示采集和分析结果”转回“把公开平台用户加工成可跟进潜在客户”。

本次不扩展运维看板、不新增复杂基础设施、不重做 CRM。目标是在现有原始数据层上增加最少必要业务对象，先跑通小红书 KET/PET 场景的真实获客闭环。

## 目标闭环

```text
帖子/评论
→ 识别是否存在真实需求
→ 关联公开用户身份
→ 合并该用户的多条内容和评论
→ 生成潜在客户卡片
→ 判断信息完整度和意向阶段
→ 进入待完善信息队列
→ 展示在获客看板
```

## 业务对象

### leads

`leads` 是按 `platform + public_profile_id` 聚合的潜在客户。每个公开用户最多对应一个当前有效 lead，后续采集会更新同一 lead，而不是重复创建客户。

字段至少包括：

- 平台用户身份
- 地区
- 需求类型
- 课程或产品
- 意向阶段
- 意向分数
- 信息完整度
- 已掌握信息
- 缺失信息
- 推荐下一步
- 当前状态：`new`、`needs_enrichment`、`qualified`、`handled`、`ignored`

### lead_evidence

`lead_evidence` 保存 lead 判断依据，来自帖子或评论。证据必须可追溯到 `content_id` 或 `comment_id`，并保留原始证据文本、需求类型、意向阶段和分数贡献。没有明确证据的用户不得生成 lead。

### enrichment_tasks

`enrichment_tasks` 保存待完善的信息和后续任务，例如：

- 补地区
- 补公开联系方式
- 确认课程或产品
- 确认年级或备考阶段
- 人工确认需求真实性

任务按 `lead_id + task_type` 幂等创建，避免每次重跑都生成重复待办。

## 生成策略

本次采用“历史回填 + 后续增量更新”：

1. 新增一个 lead 生成服务，直接读取历史 `contents`、`comments` 和 `public_profiles`。
2. 使用现有 `demand_chain` 规则识别真实需求事件。
3. 增加 KET/PET 场景规则，识别课程或产品、需求类型、意向阶段和分数。
4. 按公开用户合并多条帖子和评论，生成或更新一个 lead。
5. 为每条有效需求文本创建证据。
6. 根据缺失信息创建 enrichment task。
7. Pipeline Runner 后续每轮分析后调用同一服务，只处理本轮相关用户。

## 状态规则

- `new`：今日首次发现，有证据，但尚未判断是否完整。
- `needs_enrichment`：有真实需求，但缺少地区、联系方式、课程产品或阶段等关键跟进信息。
- `qualified`：有明确需求证据，意向分数达标，且信息完整度达到可跟进阈值。
- `handled`：人工已处理。
- `ignored`：人工忽略或判定无效。

自动生成只会把 lead 置为 `new`、`needs_enrichment` 或 `qualified`。如果用户已人工置为 `handled` 或 `ignored`，重跑不得覆盖人工状态。

## 获客页面

新增 `/leads` 作为产品主页面。默认展示：

- 今日新发现
- 待完善信息
- 可跟进客户
- 已处理客户

每张潜在客户卡片展示：

- 平台用户身份、昵称、主页链接
- 地区
- 需求类型
- 课程或产品
- 意向阶段
- 意向分数
- 信息完整度
- 已掌握信息
- 缺失信息
- 判断证据
- 推荐下一步
- 当前状态

`/ops` 保留为运维页面。聚类、查询评分和内容洞察继续保留在后台，只用于帮助系统找到更多高质量线索，不作为主页面结果展示。

## 验收标准

新的验收标准是：

- 找到多少潜在客户
- 每个客户是否有明确证据
- 多少进入待完善队列
- 多少已经达到可跟进条件

测试必须覆盖：

- 历史帖子和评论能生成 lead。
- 同一公开用户多条内容合并成一个 lead。
- 证据能追溯到帖子或评论。
- 缺失信息会生成待完善任务。
- 信息完整且意向强的 lead 会进入可跟进状态。
- 人工 `handled` / `ignored` 状态不会被自动重跑覆盖。
