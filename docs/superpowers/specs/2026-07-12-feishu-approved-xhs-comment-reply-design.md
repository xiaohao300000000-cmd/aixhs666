# 飞书审批后真实回复小红书评论设计

## 背景

当前系统已具备潜在客户筛选、飞书人工审核、私信话术草稿审批，以及小红书浏览器自动化基础。现有私信审批回调只把话术置为 `approved_to_send`，不会在飞书回调线程中直接发送。

本功能新增一条独立的评论回复闭环：针对已识别为有效潜在客户、且来源为小红书评论的线索，生成一条有帮助的公开回复草稿。人工可在飞书卡片中编辑并点击“确认发送”，系统随后立即回复目标评论，并把发送结果同步回飞书卡片和客户跟进表。

评论回复的业务目的，是先公开提供有效帮助，再在上下文合适时柔和提示对方可以私信沟通具体情况。系统不得使用强营销、索取微信或电话、夸大效果等表达。

## 范围

首版只支持：

- 平台：小红书。
- 动作：回复一条明确的目标评论。
- 目标来源：`LeadScreeningResult.source_entity_type == "comment"`，并且能关联到本地 `comments` 记录和所属帖子。
- 触发方式：人工在飞书审批卡片点击“确认发送”后立即尝试真实发送。
- 失败处理：同一卡片展示失败原因并允许人工重试。
- 状态同步：飞书审批卡片和客户跟进表均展示当前状态。

首版不支持：

- 在帖子下发布无明确回复对象的一级评论。
- 自动批量评论或无需人工审批的评论。
- 成功后再次回复同一个目标评论。
- 自动识别对方已通过私信联系并更新状态；该能力作为后续独立功能。
- 绕过登录、验证码、平台权限或平台风控限制。

## 用户流程

```text
有效评论线索
→ AI 生成评论回复草稿
→ 飞书卡片展示上下文和可编辑草稿
→ 人工点击“确认发送”
→ 系统原子领取发送权
→ 立即调用小红书浏览器回复目标评论
→ 成功或失败结果写入数据库
→ 更新原飞书卡片
→ 更新客户跟进表当前状态
```

## 文案策略

生成的回复采用“提供帮助 + 可选柔和引导”的组合：

1. 首先针对目标评论和原帖上下文给出一条简短、具体且有用的回应。
2. 只有上下文合适时，结尾才加入类似“如果方便，可以私信聊聊具体情况”的柔和提示。
3. 回复应像正常公开交流，不使用批量营销口吻。
4. 不索取手机号、微信、家庭住址等个人信息。
5. 不承诺考试结果、提分幅度、录取结果或其他无法保证的效果。
6. 不虚构机构资质、案例、价格或服务能力。
7. 草稿为空、超过平台允许长度或只包含营销引导时，不得进入发送状态。

首版生成提示应要求模型输出结构化 JSON，并限制回复为一条短文本。人工编辑后的最终文本仍需执行非空、长度和基础禁用表达校验。

## 数据模型

新增独立业务对象 `LeadCommentReply`，不复用 `LeadOutreachMessage`。评论和私信具有不同的目标定位、发送机制及客户状态，分表可以避免后续状态含义混淆。

建议字段：

- `id`
- `screening_result_id`：关联通过人工审核的筛选结果。
- `lead_id`：可空，关联客户跟进记录，便于同步客户状态。
- `target_comment_id`：关联本地目标评论。
- `target_platform_comment_id`：小红书评论 ID 的发送时快照。
- `target_content_id`：关联本地帖子。
- `target_platform_content_id`：小红书帖子 ID 的发送时快照。
- `target_url`：目标帖子 URL 快照。
- `draft_text`：AI 初始草稿。
- `approved_text`：人工确认后的最终文本。
- `status`
- `model_name`
- `feishu_chat_id`
- `feishu_message_id`
- `feishu_card_status`
- `approved_by`
- `approved_at`
- `attempt_count`
- `last_attempt_at`
- `sent_at`
- `platform_reply_id`：平台可提供时保存。
- `platform_response_json`：保存最小必要的平台响应证据，不保存登录密钥。
- `last_error`
- `created_at`
- `updated_at`

数据库约束：

- `screening_result_id` 唯一，保证同一审核结果只生成一张当前评论审批卡。
- `target_comment_id` 上建立“成功发送唯一”约束不能简单依赖普通唯一索引，因为失败记录允许重试；业务层和发送状态领取共同保证每个目标评论最多一条成功回复。
- 创建和发送操作必须支持重复调用而不重复创建、重复发布。

## 状态机

评论回复状态：

- `pending_review`：草稿已生成，等待人工审核。
- `sending`：某次飞书确认或重试已领取发送权，正在调用平台。
- `sent`：平台明确返回成功，禁止再次发送。
- `failed`：平台明确返回失败或发送前失败，允许人工重试。
- `result_unknown`：请求可能已到达平台，但客户端未能确认结果。禁止直接重试，必须先人工核对平台页面。

状态流转：

```text
pending_review → sending → sent
                         → failed → sending
                         → result_unknown
```

`sent` 为终态。`result_unknown` 不提供普通“重试”按钮，只提供“已人工确认未发送，允许重试”的恢复操作，避免网络超时后实际已发布却再次发送。

## 飞书审批卡片

卡片至少展示：

- 客户昵称或公开身份。
- 原帖标题和链接。
- 目标评论原文。
- 客户需求摘要。
- AI 判断理由。
- 可编辑的评论回复文本。
- 当前发送状态和最后一次错误。

按钮行为：

- `pending_review`：显示“确认发送”。
- `sending`：按钮禁用，显示“发送中”。
- `sent`：按钮禁用，显示“评论成功”和发送时间。
- `failed`：显示“重新发送”，保留人工编辑后的文本。
- `result_unknown`：显示“结果待确认”，不直接提供重试。

飞书回调必须继续校验 verification token、回调动作名称、记录 ID 和消息上下文。重复的同一回调以及用户连续点击不得触发第二次平台请求。

## 立即发送与幂等控制

点击确认后立即发送，但不能在数据库事务中长时间持有锁等待浏览器完成。采用三阶段流程：

1. **领取发送权**：短事务通过条件更新将 `pending_review` 或 `failed` 原子改为 `sending`，增加 `attempt_count`，保存最终文本和审核人。只有一个请求能成功领取。
2. **平台发送**：事务外调用小红书浏览器 adapter 回复目标评论。
3. **完成记录**：新事务根据平台结果把记录更新为 `sent`、`failed` 或 `result_unknown`，随后更新飞书卡片和客户跟进表。

如果飞书重复回调看到：

- `sending`：返回处理中，不再次调用平台。
- `sent`：返回已发送，不再次调用平台。
- `result_unknown`：返回待人工核对，不再次调用平台。
- `failed`：只有明确的重试动作才能重新领取发送权。

## 小红书评论发送适配器

在小红书平台适配层新增明确接口，例如：

```python
class XiaohongshuCommentReplySender(Protocol):
    def reply_to_comment(
        self,
        *,
        content_id: str,
        comment_id: str,
        text: str,
        target_url: str,
    ) -> CommentReplySendResult:
        ...
```

发送器使用现有持久化登录浏览器配置，不在核心业务层暴露 DOM selector、cookie 或平台特定响应字段。平台实现负责：

- 打开或定位目标帖子。
- 定位准确的目标评论。
- 填写回复文本并执行一次发布。
- 尽可能通过平台响应、页面新回复或其他可靠证据确认成功。
- 将结果分类为明确成功、明确失败或结果不确定。
- 登录失效、验证码、目标评论不存在或页面结构缺失时停止并返回可见错误，不尝试绕过。

首版真实发送采用当前可工作的 Playwright 浏览器路径；MediaCrawler 仍用于采集，不承担评论发布。

## 飞书多维表格客户跟进工作台

客户跟进表使用飞书多维表格实现，作为运营人员日常查看、筛选、分配和维护客户进度的工作台。本地 PostgreSQL 仍是评论发送事实、幂等控制和审计记录的事实源；多维表格不独立执行真实评论发送，也不替代本地发送状态机。

评论回复记录应关联 `LeadScreeningResult`、公开用户和对应 `Lead`。每次状态变化后同步多维表格客户记录：

| 评论状态 | 客户当前状态 |
|---|---|
| `pending_review` | `评论待审核` |
| `sending` | `评论发送中` |
| `sent` | `已评论引导，等待客户私信` |
| `failed` | `评论发送失败` |
| `result_unknown` | `评论结果待确认` |

跟进表还应展示最近评论时间、最近错误和评论回复记录 ID，便于从客户记录定位审批卡片。评论状态同步不得覆盖 `已收到私信`、`已成交`、`已忽略` 等更后置或人工锁定的客户状态。

如果数据库状态已更新但飞书同步失败，数据库发送结果仍为事实来源，并保存同步错误以供后续重试。不得因为飞书更新失败而再次发布评论。

### 建议字段

客户跟进表至少包含：

- 客户唯一键：系统生成，只读，用于幂等 upsert。
- 客户昵称、公开主页、地区、需求摘要、意向等级。
- 原帖链接、目标评论原文。
- 当前客户状态。
- 负责人。
- 运营备注。
- 下次跟进时间。
- 评论审批状态：系统只读。
- 评论发送结果：系统只读。
- 最近评论时间：系统只读。
- 最近评论错误：系统只读。
- 评论回复记录 ID：系统只读。
- 审批卡片消息 ID 或可定位信息：系统只读。

### 字段写入权限

运营人员可以在多维表格中修改：

- 当前客户状态中的人工业务状态，例如 `已收到私信`、`沟通中`、`已成交`、`已忽略`。
- 负责人。
- 运营备注。
- 下次跟进时间。

系统负责写入并覆盖：

- 评论审批状态。
- 评论发送结果。
- 最近评论时间。
- 最近评论错误。
- 评论回复记录 ID。
- 平台发送证据和其他系统事实字段。

系统事实字段在多维表格中应通过字段权限或专用视图尽量设置为运营只读。即使运营人员通过其他方式修改了显示值，下一次系统同步也以本地数据库事实覆盖。

### 双向同步规则

- 本地到飞书：按客户唯一键执行幂等 upsert，同步客户基础信息和系统事实字段。
- 飞书到本地：只拉取允许人工维护的业务字段，不接受飞书覆盖评论发送事实。
- 人工客户状态不得被较早的自动状态回退。例如已经是 `已收到私信` 或 `已成交`，后续评论同步不得改回 `已评论引导，等待客户私信`。
- 对同一字段同时修改时，系统事实字段始终以本地数据库为准；人工业务字段以飞书最近一次有效修改为准，并记录同步时间。
- 同步失败只记录和重试同步，不触发评论重新发送。

### 推荐视图

首版建议创建以下多维表格视图：

- `评论待审核`：等待人工确认回复文本。
- `评论发送失败`：明确失败且允许重试。
- `评论结果待确认`：可能已发布，需要人工核对平台。
- `等待客户私信`：评论成功但尚未记录客户主动私信。
- `沟通中`：运营已确认收到私信或进入沟通。
- `今日待跟进`：下次跟进时间为今天或已逾期。
- `已成交/已忽略`：终态客户归档。

## 可视化看板

可使用飞书多维表格自带仪表盘，或项目的可视化工具构建只读运营看板。可视化层只读取本地数据库或多维表格同步结果，不直接修改评论状态，不执行发送，也不维护第二套客户主数据。

建议展示：

- 评论草稿数、待审核数、发送成功数、发送失败数和结果待确认数。
- 从“有效线索 → 评论成功 → 收到私信 → 沟通中 → 成交”的转化漏斗。
- 每日评论成功量和客户主动私信量趋势。
- 按负责人、需求类型、地区和意向等级的客户分布。
- 失败原因分类和需要人工处理的异常列表。

首版优先使用飞书多维表格视图和仪表盘，减少额外部署和权限维护。只有当飞书原生仪表盘不能满足交互或分析需求时，再增加项目内只读可视化页面。

## 错误处理

错误分为：

- **发送前明确失败**：登录失效、目标记录缺字段、目标评论不存在、文本校验失败。进入 `failed`，允许修复后重试。
- **平台明确拒绝**：平台返回失败、频率限制或内容限制。进入 `failed`，展示平台可安全公开的错误摘要。
- **结果不确定**：点击发布后发生超时、浏览器崩溃或响应证据不足。进入 `result_unknown`，禁止普通重试。
- **飞书同步失败**：发送状态照常落库，记录同步错误并允许独立重试飞书同步。

日志和数据库不得记录 cookie、Authorization、飞书 App Secret 或完整浏览器存储状态。

## 安全与运营限制

- 所有真实评论必须经过一次明确的人工确认。
- 默认不提供批量确认发送。
- 每个目标评论最多一条成功回复。
- 首版同一时间只执行一个真实评论发送，降低账号操作冲突。
- 发送器发现验证码、登录失效或平台限制后立即停止，不自动破解或高频重试。
- 评论回复必须保留目标评论、最终文本、审核人、发送时间和结果证据，满足审计需要。
- 后续如增加限频，应作为配置项和独立决策记录，不在首版默认为高频自动化。

## 测试策略

自动测试覆盖：

- 只有有效且来源为评论的筛选结果能创建回复草稿。
- 同一筛选结果重复创建不会产生第二条记录或第二张卡片。
- 卡片展示正确目标评论和可编辑文本。
- 未确认时不会调用真实发送器。
- 确认后从 `pending_review` 原子进入 `sending`，并调用发送器一次。
- 重复确认、重复回调和并发领取不会调用发送器两次。
- 明确成功进入 `sent`，保存发送时间和平台证据。
- 明确失败进入 `failed`，飞书卡片显示错误并允许重试。
- 失败重试成功后进入 `sent`，且总成功发布次数为一。
- 结果不确定进入 `result_unknown`，普通重试被拒绝。
- 飞书卡片更新失败不改变已记录的平台发送结果。
- 客户跟进表按状态同步，且不覆盖更后置或人工锁定状态。
- 多维表格允许修改负责人、运营备注、下次跟进时间和人工业务状态，并能安全同步回本地数据库。
- 多维表格不能覆盖评论是否已发送、发送时间和平台结果等系统事实。
- 文本为空、过长或包含禁止表达时拒绝发送。

真实平台测试必须使用一条专门准备的测试帖子和测试评论，并由人工在飞书确认。测试完成后记录目标、时间、最终文案和平台结果，但不把账号凭据写入仓库。

## 验收标准

- 有效评论线索可以生成一张包含上下文和可编辑回复的飞书审批卡片。
- 未点击确认前，小红书不会产生任何评论回复。
- 点击“确认发送”后，系统立即尝试回复准确的目标评论。
- 成功后卡片显示“评论成功”，客户跟进表显示“已评论引导，等待客户私信”。
- 明确失败后卡片显示失败原因和“重新发送”，客户跟进表显示“评论发送失败”。
- 运营人员可在飞书多维表格中维护负责人、备注、下次跟进时间及后置客户状态。
- 多维表格中的评论发送事实由系统只读维护，人工修改不能导致数据库状态变化或重复发送。
- 飞书视图或只读可视化看板能展示评论发送状态和客户转化漏斗。
- 发送结果不确定时不会自动重试或重复评论。
- 重复飞书回调、重复点击和重复执行不会产生第二条成功回复。
- 所有关键状态、最终文本、审核人、尝试次数和发送结果均可追溯。
