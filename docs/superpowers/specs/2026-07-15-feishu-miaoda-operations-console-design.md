# 飞书工作台 + 妙搭运营控制台设计

日期：2026-07-15  
状态：待用户审阅  
目标分支：`feat/v16-task-productization`

## 1. 背景

当前项目已经具备：

- PostgreSQL 事实源；
- FastAPI API；
- 持久任务与 Python Worker；
- 小红书采集适配器；
- AI 线索筛选、Campaign 资格判断和飞书同步；
- `skill_runs` / `skill_run_events`；
- 飞书 Card 2.0 任务中心原型；
- `/leads`、飞书多维表格和系统控制台等运营入口。

现状问题不是缺少业务能力，而是运营入口分散：

- 飞书卡片适合通知、审批和快捷任务，但不适合复杂图表、配置、批量操作和排障；
- 现有 Web 页面缺少统一的产品结构与日常工作台；
- Campaign、任务、审核、客户和系统健康没有形成同一个可视化操作面；
- `localtunnel` 只适合开发验收，不能作为稳定运营入口；
- 如果直接把全部业务迁移到妙搭，会重复现有 Python、PostgreSQL 和 Worker 能力，并引入较大回归风险。

本设计选择：

> 妙搭全栈应用作为主控制台，飞书作为快捷入口和协作层，现有 FastAPI、PostgreSQL、Skill Runtime 和 Worker 继续作为业务内核。

## 2. 产品目标

### 2.1 核心目标

提供一个每天可使用的可视化运营台，让运营人员能够：

1. 打开后立即看到今日待办、系统异常和正在运行的任务；
2. 连续审核 AI 发现的线索，并查看完整证据；
3. 从模板创建任务、预览影响范围、确认运行并追踪结果；
4. 配置多个行业、客户和 Campaign，而不修改业务代码；
5. 从飞书收到提醒、完成快捷操作，并一键进入对应控制台页面；
6. 看见 API、Worker、飞书连接和数据新鲜度是否健康。

### 2.2 业务目标

第一阶段用于内部多个业务，同时支持客户定制交付，并保留未来产品化空间。

效果验证目标：

- 四周内接入至少两个业务场景；
- 产出至少 50 条人工确认后可继续跟进的客户；
- 建立采集候选 → 有效需求 → 高意向 → 可跟进 → 已跟进的完整漏斗；
- 每条客户可追溯到原始内容、需求证据、AI 判断、人工结果和 Campaign 配置版本。

## 3. 非目标

第一版不做：

- 标准 SaaS 计费、套餐和自助开通；
- 任意代码插件市场；
- 所有业务能力迁移到妙搭；
- 妙搭数据库复制线索、客户、任务和 Campaign 事实数据；
- Redis、Kafka、Kubernetes、Temporal 或新任务队列；
- 在飞书卡片里完成所有高级配置；
- 一次性重写现有 Web、Worker 或采集体系；
- 让 AI 直接决定真实外联或高风险写操作。

## 4. 用户与权限

第一版定义四类角色：

### 4.1 运营人员

- 查看今日工作台；
- 审核线索；
- 运行已发布的安全任务模板；
- 查看任务进度与结果；
- 分配客户和更新跟进状态。

### 4.2 业务管理员

- 拥有运营人员权限；
- 创建和修改客户配置与 Campaign；
- 测试和发布 Campaign 版本；
- 配置负责人、业务范围和通知策略。

### 4.3 系统管理员

- 拥有业务管理员权限；
- 查看系统健康、Trace 和错误；
- 处理权限、Worker、集成和环境配置；
- 执行明确确认后的高风险恢复操作。

### 4.4 只读观察者

- 查看总览、漏斗、任务结果和客户状态；
- 不执行审核、运行、发布或重试。

身份以飞书用户为入口，后端将 `tenant_key`、`open_id` 映射到内部 operator 和角色。所有写操作记录 operator、request_id、资源 ID、变更前后值和时间。

## 5. 产品结构

### 5.1 妙搭主控制台

主导航：

1. 今日工作台；
2. 线索中心；
3. 审核工作台；
4. 任务运行；
5. Campaign；
6. 数据分析；
7. 系统健康。

### 5.2 飞书工作台

飞书负责：

- 新线索、待审核、任务完成和系统异常提醒；
- 运行预设任务；
- 快速批准、拒绝、观察和分配；
- 查看任务摘要；
- 深度链接到妙搭中的具体线索、任务、Campaign 或异常。

飞书不负责：

- 复杂 Campaign 配置；
- 大规模批量操作；
- 完整历史分析；
- 高密度系统排障；
- 编辑原始 Prompt 或配置文档。

## 6. 今日工作台

首页采用日常操作优先，而不是分析优先。

### 6.1 信息优先级

从上到下：

1. 阻断业务的系统异常；
2. 超时或高价值待办；
3. 正在运行和需要确认的任务；
4. 快捷任务入口；
5. 今日漏斗和结果指标。

### 6.2 主要待办

- 高意向线索等待审核；
- 客户等待分配；
- 跟进超过时限；
- 任务失败或结果不确定；
- Campaign 草稿等待测试或发布；
- Worker、飞书、数据库或采集器异常。

### 6.3 快捷任务

第一版至少支持：

- 历史线索智能筛选；
- 同步审核结果；
- 分配待跟进客户；
- 打开完整任务中心。

## 7. 线索审核工作台

采用“左侧队列 + 右侧证据卡”的 A+B 组合布局。

### 7.1 左侧队列

展示：

- 标题或需求摘要；
- AI 初始判断；
- Campaign；
- 等待时间；
- 超时、重复或异常状态。

支持：

- 按业务、意向、等待时间和异常筛选；
- J/K 或上下键切换；
- 处理后自动进入下一条；
- 显示剩余数量和预计处理时间。

### 7.2 右侧证据卡

必须同屏展示：

- 原始内容和来源链接；
- 作者与发布时间；
- 需求、地域、阶段、时间和预算等结构化证据；
- AI 判断理由；
- 评分明细；
- Campaign 命中与排除规则；
- 客户历史、重复记录和已有跟进；
- 使用的 Campaign 配置版本。

### 7.3 审核动作

- 确认有效；
- 暂时观察；
- 无效；
- 分配负责人；
- 标记需要补充信息；
- 合并重复客户。

无效或观察必须记录标准原因，例如：

- 没有真实购买意图；
- 地域不匹配；
- 只有泛讨论；
- 同行推广；
- 信息过期；
- 证据不足；
- 无法继续联系；
- 重复客户；
- Campaign 不匹配。

人工结果写入 PostgreSQL，并作为后续 Prompt、规则和评分校准的真实标注集。

## 8. 任务中心

采用“任务模板 + 中间运行历史 + 右侧配置和预览”。

### 8.1 任务模板

模板来自 Skill Registry，但产品层只展示已发布、可用且符合当前角色的模板。

第一版模板：

- 历史线索智能筛选；
- 同步审核结果；
- 运行候选采集；
- 分配待跟进客户。

### 8.2 参数填写

任务只展示需要的安全参数，例如：

- 数据范围；
- Campaign；
- 处理数量；
- 来源类型；
- 预算；
- 筛选模式。

高级或危险参数不直接暴露给普通运营人员。

### 8.3 运行预览

执行前必须展示：

- 符合条件的候选数量；
- 将读取和写入的数据范围；
- 预计耗时；
- 预计模型或外部调用成本；
- 是否访问外部平台；
- 是否包含不可中断阶段；
- 是否允许取消和重试。

### 8.4 运行状态

统一状态：

```text
draft
→ previewed
→ queued
→ running
→ succeeded | failed | cancelled | result_uncertain
```

每个状态变化写入 `skill_run_events`。

### 8.5 恢复策略

- 模型超时、临时网络失败、幂等同步失败：可安全重试；
- 外部写入结果不确定：进入 `result_uncertain`，需要人工确认；
- 权限变化、配置版本失效、数据合同变化：禁止自动继续；
- Worker 重启：从持久 checkpoint 恢复；
- 重复点击：使用幂等键复用现有 Run 或返回当前状态。

## 9. Campaign 配置中心

采用三级继承：

```text
行业模板
→ 客户配置
→ Campaign 覆盖
```

### 9.1 行业模板

定义：

- 通用需求信号；
- 通用排除条件；
- 默认评分权重；
- 默认 Prompt；
- 通用跟进字段。

### 9.2 客户配置

覆盖：

- 品牌与业务范围；
- 产品和服务方式；
- 可服务地域；
- 特有排除条件；
- 客户字段与负责人策略；
- 通知和同步目标。

### 9.3 Campaign 覆盖

覆盖：

- 本次目标产品；
- 地域；
- 时间范围；
- 查询词与来源；
- 预算和数量；
- 当前优先级；
- 本次高意向阈值。

### 9.4 配置版本

每次发布生成不可变版本，记录：

- 基础版本；
- 覆盖内容；
- 发布人；
- 发布时间；
- 发布原因；
- 测试样本结果；
- 回滚来源。

已运行的 Skill Run 必须引用明确配置版本，后续配置修改不能改变历史结果解释。

### 9.5 样本测试

发布前使用真实正负样本测试：

- 结构化字段提取；
- 命中和排除规则；
- 分数变化；
- 最终判断；
- 与人工标签的差异。

未经测试的草稿不能直接发布为线上配置。

## 10. 技术架构

```text
飞书工作台 ─┐
             ├─ HTTPS Operator API ─ FastAPI 业务内核
妙搭控制台 ─┘                          │
                                      ├─ PostgreSQL
                                      ├─ 对象存储
                                      ├─ Skill Runtime
                                      ├─ Lead Engine
                                      ├─ Campaign Engine
                                      ├─ Auth & Audit
                                      └─ Integration Gateway
                                               │
                                               ├─ Python Worker
                                               ├─ 采集 Worker
                                               ├─ AI 模型
                                               └─ 飞书 OpenAPI
```

### 10.1 妙搭职责

- 页面和交互；
- 飞书身份接入；
- 调用 Operator API；
- 保存界面偏好和必要临时会话；
- 展示服务端返回的业务状态；
- 不实现核心业务流程；
- 不复制业务事实数据。

### 10.2 FastAPI 职责

- 统一鉴权；
- 查询和写入业务数据；
- 创建、预览、取消和重试 Skill Run；
- Campaign 配置计算、测试和发布；
- 审核、分配与跟进；
- 返回系统健康和运行事件；
- 生成飞书深度链接目标。

### 10.3 PostgreSQL 职责

唯一保存：

- 原始与标准化数据；
- 线索、客户和证据；
- Campaign 与配置版本；
- Skill Run 与事件；
- 人工审核与跟进；
- operator、角色和审计事件。

### 10.4 稳定部署

生产入口必须使用稳定 HTTPS 域名，替换 `localtunnel`。

最低要求：

- API 和 Worker 进程守护；
- 健康检查；
- 请求 Trace；
- 结构化错误日志；
- 环境变量安全管理；
- 数据库备份；
- 外部依赖超时和重试；
- 发布和回滚流程。

## 11. Operator API 边界

第一版 API 资源：

- `/operator/dashboard`：今日待办、运行和核心指标；
- `/operator/leads`：线索列表与筛选；
- `/operator/leads/{id}`：证据、判断、历史和配置版本；
- `/operator/leads/{id}/review`：人工审核；
- `/operator/leads/{id}/assign`：分配负责人；
- `/operator/skills`：可用任务模板；
- `/operator/skill-runs/preview`：任务预览；
- `/operator/skill-runs`：创建和查询 Run；
- `/operator/skill-runs/{id}/cancel`：取消；
- `/operator/skill-runs/{id}/retry`：明确重试；
- `/operator/campaigns`：Campaign 列表；
- `/operator/campaigns/{id}/draft`：草稿；
- `/operator/campaigns/{id}/test`：样本测试；
- `/operator/campaigns/{id}/publish`：发布；
- `/operator/health`：API、Worker、数据库、飞书和数据新鲜度。

接口只返回明确合同，不把 SQLAlchemy 模型或飞书原始字段直接泄漏给前端。

## 12. 飞书集成

### 12.1 通知

- 高价值线索等待审核；
- 审核超时；
- 任务完成；
- 任务失败或结果不确定；
- Worker、API 或飞书连接异常；
- Campaign 草稿等待发布。

### 12.2 快捷操作

- 确认有效；
- 暂时观察；
- 无效；
- 分配负责人；
- 运行安全任务模板；
- 打开对应妙搭页面。

### 12.3 深度链接

每条飞书消息携带明确资源定位：

- lead_id；
- skill_run_id；
- campaign_id；
- alert_id。

妙搭打开后直接定位资源，不让用户再次搜索。

## 13. 错误处理

### 13.1 用户可恢复错误

页面必须展示：

- 发生了什么；
- 哪一步失败；
- 数据是否已经保存；
- 是否可以安全重试；
- 推荐恢复动作；
- request_id 和 run_id。

### 13.2 系统错误

禁止只显示“操作失败”。错误按类型归类：

- 权限；
- 配置；
- 数据校验；
- 外部依赖；
- 超时；
- 结果不确定；
- 内部异常。

### 13.3 降级

- 飞书通知失败：业务状态仍保存，操作台显示待补发；
- 妙搭暂时不可用：Worker 和业务任务继续运行；
- AI 模型不可用：任务进入可恢复失败，不产生虚假结果；
- 采集器不可用：历史数据审核和筛选仍可使用；
- 系统健康接口异常：控制台显示未知，不伪装为正常。

## 14. 可观测性

操作台系统健康至少显示：

- API 状态与延迟；
- Worker 最后心跳；
- 队列积压数量；
- 最近成功和失败任务；
- PostgreSQL 连通性；
- 最后采集、筛选和飞书同步时间；
- 飞书 user/bot 身份和权限状态；
- 当前发布的 Campaign 版本；
- 外部 AI 模型最近错误率。

所有写操作贯穿：

```text
operator_id
request_id
resource_id
skill_run_id（如适用）
campaign_version_id（如适用）
```

## 15. 测试与验收

### 15.1 自动测试

覆盖：

- Operator API 鉴权和角色；
- 审核状态转换；
- 审核幂等；
- 重复点击；
- 任务预览；
- 创建、取消、失败和重试；
- Worker 重启恢复；
- Campaign 三级继承；
- 配置版本不可变；
- 样本测试和发布门禁；
- 飞书消息深度链接；
- 飞书失败不影响事实状态；
- 系统健康降级；
- 审计事件完整性。

### 15.2 真实验收

必须使用：

- 真实飞书身份；
- 真实妙搭应用；
- 真实 PostgreSQL；
- 真实 61 条存量线索；
- 真实 Worker；
- 稳定 HTTPS API。

验收流程：

1. 飞书收到待审核提醒；
2. 点击后打开妙搭指定线索；
3. 查看完整证据并完成审核；
4. PostgreSQL 更新审核和审计事件；
5. 今日待办数量变化；
6. 从模板创建历史线索筛选任务；
7. 预览后确认执行；
8. Worker 更新进度；
9. 完成后飞书收到结果；
10. 任务结果和线索列表保持一致。

## 16. 四阶段上线

### 阶段 1：基础与只读总览

范围：

- 妙搭应用；
- 飞书登录；
- 稳定 Operator API；
- 今日工作台只读数据；
- 系统健康和 Trace。

验收：真实 PostgreSQL 数据可见，错误可定位。

### 阶段 2：线索审核闭环

范围：

- A+B 组合审核工作台；
- 有效、观察、无效；
- 拒绝原因；
- 分配负责人；
- 飞书提醒与深度链接。

验收：61 条存量线索可以完整审核和追溯。

### 阶段 3：任务中心闭环

范围：

- 任务模板；
- 参数与运行预览；
- 创建、进度、取消、失败和重试；
- 飞书任务通知。

验收：历史筛选任务通过控制台真实运行并产生一致结果。

### 阶段 4：Campaign 与多业务

范围：

- 三级配置继承；
- 样本测试；
- 版本发布与回滚；
- 第二业务场景；
- Campaign 漏斗对比。

验收：两个业务共享核心流程，使用不同配置，并形成可比较漏斗。

## 17. 迁移策略

### 17.1 保留

- PostgreSQL 数据模型和历史数据；
- FastAPI 核心业务服务；
- Worker 和任务领取机制；
- Skill Registry 和 Skill Runtime；
- 飞书 OpenAPI 适配层；
- 现有测试和 V16 真实验收成果。

### 17.2 新增

- 妙搭全栈应用；
- Operator API；
- operator/role/audit 能力；
- Campaign 模板、客户覆盖和版本测试能力；
- 稳定部署和系统健康；
- 飞书到妙搭的深度链接。

### 17.3 逐步替代

- `/leads` 逐步由妙搭线索中心替代；
- 飞书原生任务卡保留为快捷入口，不再承担全部管理功能；
- `localtunnel` 仅保留开发调试，不作为验收或生产入口；
- 多维表格保留为协作视图，不作为事实源。

## 18. 关键决策摘要

1. 方案 2 与方案 3 结合：妙搭为主控制台，飞书为入口和协作层；
2. 首页采用日常操作优先的今日工作台；
3. 审核采用左队列 + 右证据卡；
4. 任务中心采用模板 + 右侧配置和预览；
5. Campaign 采用行业模板 → 客户配置 → Campaign 覆盖；
6. 保留现有 FastAPI、PostgreSQL 和 Worker，不迁移核心；
7. 使用四阶段渐进上线，每阶段独立真实验收；
8. 飞书、妙搭、CLI 和 Worker 共享同一事实源和审计链路。
