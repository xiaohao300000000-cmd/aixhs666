# Agent + 飞书客户跟进表设计

## 背景

项目方向从“网页看板作为主产品入口”调整为“运行框架 + agent + 飞书多维表格”。现有 `/leads` 和 `/ops` HTML 页面保留为可选本地入口和调试工具，但不再作为主要使用界面。

第一版使用 PostgreSQL 作为系统主数据库，飞书多维表格作为用户日常工作台。系统内部继续保留 `leads`、`lead_evidence`、`enrichment_tasks`、`pipeline_runs` 等结构化对象，用于去重、追溯、幂等和后续分析。飞书侧只展示一张面向用户的“客户跟进表”，不暴露内部专业名词。

## 目标

1. 让普通用户只在飞书里看到一张能直接跟进的客户表。
2. 让 agent 自动完成运行调度、客户筛选、优先级排序和飞书同步。
3. 把飞书人工状态、负责人和备注回写到 PostgreSQL，避免重复推送。
4. 改进需求识别方式，减少广告、老师、攻略、无需求讨论被误判成客户。
5. 保留网页和 API 作为调试入口，但不把它们作为主工作流。

## 非目标

- 不删除现有 HTML 页面。
- 不做自动私信、自动评论或自动成交。
- 不把飞书多维表格作为主数据库。
- 第一版不拆多张飞书表，不让用户看到“证据表”“运行表”“补充任务表”等内部结构。
- 第一版不允许 agent 自动修改代码或自动改写识别规则。

## 总体架构

```text
定时器 / 手动命令
  -> AgentRuntime
  -> PipelineRunner
  -> 采集、分析、lead 生成
  -> AgentLeadPrioritizer
  -> FeishuBitableSync
  -> 飞书“客户跟进表”
  -> 用户修改状态、负责人、备注
  -> FeishuFeedbackSync
  -> PostgreSQL
```

### PostgreSQL

PostgreSQL 仍是主数据库。它负责：

- 采集内容、评论、公开用户和发现路径。
- lead 去重和历史保留。
- 需求判断证据和推荐原因追溯。
- pipeline 运行记录。
- 飞书记录 ID 映射和同步状态。
- 人工反馈回写后的系统状态。

### 飞书多维表格

飞书是用户工作台。第一版只维护一张表，建议命名为：

```text
客户跟进表
```

用户可见字段使用人话：

| 字段 | 含义 |
|---|---|
| 客户 | 小红书昵称；没有昵称时用公开用户 ID |
| 需求 | 用一句话说明这个人可能想解决什么问题 |
| 课程/考试 | KET、PET 等 |
| 意向程度 | 高 / 中 / 低 |
| 为什么推荐 | 一句话说明系统推荐原因 |
| 下一步 | 建议人工怎么跟进 |
| 状态 | 新发现 / 待确认 / 可跟进 / 已跟进 / 不合适 |
| 来源链接 | 原帖子或评论链接 |
| 发现时间 | 系统首次发现时间 |
| 负责人 | 人工填写 |
| 备注 | 人工填写 |

内部状态映射为用户语言：

| 系统状态 | 飞书显示 |
|---|---|
| new | 新发现 |
| needs_enrichment | 待确认 |
| qualified | 可跟进 |
| handled | 已跟进 |
| ignored | 不合适 |

## Agent 职责

第一版 agent 做确定性决策，不做自由发挥。

### 运行调度

agent 根据当前数据库状态选择要运行的查询：

- 优先跑最近产出有效客户的查询。
- 降低重复率高、广告噪音高、近期失败多的查询。
- 保留少量探索额度给新词或低覆盖查询。

### 客户筛选

agent 每轮只把两类客户同步到飞书：

- 可跟进客户：需求较明确，值得人工马上看。
- 待确认客户：有需求迹象，但缺地区、年级、考试时间、联系方式等信息。

明显广告、老师招生、纯攻略、资料分享、无需求讨论不进入飞书。

### 优先级排序

排序依据包括：

- 需求动作是否明确，例如问课程、问机构、问价格、问是否报班、二刷、没过。
- 是否有孩子、年级、考试、地区、机构、时间压力等上下文。
- 是否来自近期内容。
- 是否已有多条相关表达。
- 是否被人工反馈为“不合适”的相似模式命中。

### 人话生成

agent 生成给飞书看的字段：

- `需求`：一句话概括用户问题。
- `为什么推荐`：一句话说明推荐依据。
- `下一步`：一句话建议人工动作。

这些字段必须来自原始文本和结构化判断，不允许编造联系方式、地区、年级或考试时间。

## 需求识别策略

需求识别不再做单步“是不是客户”分类，改为三层识别。

### 第一层：硬排除

先用规则和轻量判断排除明显不是客户的内容：

- 老师、机构、课程顾问、招生广告。
- 资料分享、求资料、蹲链接。
- 纯经验分享，没有求助动作。
- 博主教程、备考攻略。
- 已经考完只是在晒成绩。
- 当前业务范围外的成人考试或非目标课程。

这层目标是降低误报，不负责发现客户。

### 第二层：动作意图识别

对可能有需求的文本识别动作，而不是直接判断“是不是客户”：

- 问课程。
- 问机构。
- 问价格。
- 问试听。
- 问是否需要报班。
- 表达没过、压线、二刷。
- 比较机构或课程方案。
- 求提升方法。

动作意图比“客户”更客观，适合做结构化标签。

### 第三层：上下文判断

结合上下文决定是否进飞书：

- 评论所在帖子主题。
- 用户历史相关表达。
- 孩子、年级、考试、地区、课程、机构、时间压力。
- 缺失信息是否足以放到“待确认”。
- 是否命中人工反馈过的误报模式。

最终输出固定结构：

```text
是否进入飞书：是 / 否
进入类型：可跟进 / 待确认
需求：一句话
为什么推荐：一句话
缺失信息：地区 / 年级 / 考试时间 / 联系方式 / 其他
下一步：一句话
置信度：高 / 中 / 低
不进入原因：广告 / 老师招生 / 攻略 / 资料分享 / 无明确需求 / 业务范围外 / 其他
```

低置信度但仍有真实需求迹象的内容进入“待确认”，不要直接丢弃。

## 飞书同步

### 写入飞书

系统按 `platform + public_profile_id` 或内部 `lead_id` 做幂等同步。已有飞书记录时更新，不重复创建。

写入规则：

- `可跟进` 和 `待确认` 写入飞书。
- `已跟进` 和 `不合适` 不被新一轮自动覆盖回未处理状态。
- 人工填写的 `负责人` 和 `备注` 不被系统覆盖。
- 系统只更新它负责的字段，例如需求、意向程度、为什么推荐、下一步、来源链接、发现时间。

### 读回飞书

系统定时读取飞书状态：

- `已跟进` 回写为 `handled`。
- `不合适` 回写为 `ignored`。
- `可跟进` 回写为 `qualified`。
- `待确认` 回写为 `needs_enrichment`。
- `负责人`、`备注` 回写到系统中的同步元数据或 lead 扩展字段。

读回后，agent 后续运行必须尊重人工状态。

## 数据模型调整

第一版需要新增一个飞书同步映射对象，用于记录本地对象和飞书记录的关系：

```text
feishu_bitable_records
- id
- local_entity_type
- local_entity_id
- app_token
- table_id
- record_id
- sync_direction
- last_synced_at
- last_remote_updated_at
- last_sync_status
- last_error
- created_at
- updated_at
```

可选地给 `leads` 增加人工字段：

```text
owner_name
operator_note
last_feedback_at
```

如果不改 `leads` 表，也可以把这些字段放在同步映射对象的 `remote_fields_json` 中。实现时优先选择对现有模型侵入更小的方案。

## 配置

需要新增飞书多维表格配置：

```text
FEISHU_ENABLED=true
FEISHU_APP_ID=
FEISHU_APP_SECRET=
FEISHU_BITABLE_APP_TOKEN=
FEISHU_LEADS_TABLE_ID=
FEISHU_SYNC_DRY_RUN=false
FEISHU_SYNC_PAGE_SIZE=100
```

Webhook 仍可保留给告警和交互卡片，但多维表格同步需要使用飞书开放平台 API，而不是机器人 webhook。

## 错误处理

- 飞书凭证缺失时，同步进入 dry-run 或 blocked，不影响采集和本地 lead 生成。
- 飞书写入失败时记录错误，不重复创建同一 lead。
- 飞书限流时使用退避重试。
- 字段缺失或表结构不匹配时，输出明确诊断，提示缺哪个字段。
- 飞书状态读回失败时，不覆盖本地状态。
- 人工状态优先级高于系统自动状态。

## 测试

需要覆盖：

1. 需求识别硬排除：广告、老师、攻略、资料分享不进飞书。
2. 动作意图识别：问课程、问机构、问价格、二刷、没过等能生成结构化意图。
3. 上下文判断：低置信度但有需求迹象进入“待确认”。
4. 飞书字段映射：内部字段正确翻译成人话字段。
5. 幂等同步：同一个 lead 多次同步只对应一条飞书记录。
6. 人工字段保护：负责人和备注不被系统覆盖。
7. 状态读回：飞书状态能回写到 PostgreSQL。
8. 凭证缺失和 API 失败：不会影响主流程。
9. Pipeline 集成：一轮运行后 agent 能筛选并同步可见客户。

## 验收标准

1. 用户只需要打开飞书“客户跟进表”就能看到新客户和待确认客户。
2. 表格字段没有 `lead_evidence`、`pipeline_runs`、`needs_enrichment` 等专业词。
3. 明显广告、老师招生、攻略和资料分享不会进入飞书表。
4. 用户在飞书改状态后，系统能回写并避免重复推送。
5. `pytest -q` 通过。
6. 没有真实飞书凭证时，dry-run 可以输出将要写入的记录和字段。

## 后续扩展

- 当一张表不够用时，再增加“证据明细表”或“运行记录表”，但不作为第一版范围。
- 人工反馈积累足够后，可以生成规则调整建议；建议必须人工确认后再生效。
- 可以增加 agent 每日摘要，把新增客户、误报、待确认数量推送到飞书群。
