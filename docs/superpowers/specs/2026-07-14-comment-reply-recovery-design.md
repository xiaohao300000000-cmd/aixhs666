# 评论回复中断恢复设计

## 背景

本次恢复只完成中断前已经确定的自动化与文档工作，不尝试真实小红书发送。

## 目标

- 保留并审查现有评论回复改动，不重新实现已经通过定向测试的功能。
- 让飞书回调、持久任务、Worker、远程 CDP 和运营文档描述一致。
- 完成定向测试、全量测试和 `git diff --check`。
- 只修复验证或审查发现的缺陷，不扩展评论回复功能范围。
- 明确保留 live selector probe 和单条真实发送的外部阻塞状态。

## 架构边界

### 飞书回调

飞书“确认发送”或允许重试的回调只负责：

1. 校验 verification token、操作人、卡片消息和聊天绑定。
2. 校验人工编辑后的最终回复文本。
3. 原子地将记录推进到 `approved_to_send`。
4. 创建一个 `comment_reply_send` 持久任务。
5. 快速返回 `accepted`，不构造浏览器 sender，不运行 Playwright。

重复回调只返回数据库中的当前状态，不创建第二个发送任务。

### Worker 发送

Worker 领取 `comment_reply_send` 后：

1. 条件领取 `approved_to_send` 记录并推进到 `sending`。
2. 增加 `attempt_count`，形成当前 attempt fencing。
3. 使用已批准文本调用评论发送器。
4. 持久化 `sent`、`failed` 或 `result_unknown`。
5. 更新飞书结果卡片。
6. 独立同步客户跟进表；同步失败不得触发平台重发。

任务自身只允许一次调度重试。平台结果不确定时必须停在 `result_unknown`，由人工核对。

### 浏览器连接

评论发送器支持两种显式模式：

- `local`：保留本地持久浏览器能力，用于开发和既有测试。
- `remote_cdp`：要求 Chromium 和 `COMMENT_REPLY_CDP_URL`，通过 `connect_over_cdp` 复用 Windows Chrome 的现有 context；不得在 Mac 启动本地浏览器，也不得关闭远端 context。

生产评论回复使用 `remote_cdp`。远端 CDP 不可用时任务必须明确失败并保留错误，不允许静默回退到本地浏览器。

### 页面交互与 Probe

小红书目标评论的编辑器和提交按钮只在点击“回复”后出现。因此发送和 probe 的顺序必须是：

1. 精确定位唯一目标评论容器。
2. 精确定位唯一回复按钮。
3. 点击回复按钮展开控件。
4. 等待并验证唯一编辑器和唯一提交按钮。

Selector probe 到此为止，禁止填写文本和点击提交。真实发送只有在同一次人工验收窗口内提供专用目标、最终文本和明确批准后才能执行一次。

## 文档恢复

`docs/COMMENT_REPLY_OPERATIONS.md` 需要同步以下事实：

- 自动化实现是异步持久任务，不是同步回调发送。
- 新增 `approved_to_send` 的 Base 状态映射。
- 生产浏览器由远程 Windows CDP 提供。
- 回调延迟验收改为验证快速入队和重复回调幂等，不再等待平台发送完成。
- Worker、远程 CDP 和 live 测试所需配置必须分别列明。
- 截至 2026-07-13，远程 CDP/SSH 和 live 测试参数仍阻塞，不能宣称真实发送成功。

## 验证策略

按风险从小到大验证：

1. 评论页面交互、远程 CDP 配置与 probe 单元测试。
2. 飞书回调入队、重复回调和状态机测试。
3. Worker 持久任务派发测试。
4. 评论回复相关定向测试集合。
5. 全量 `pytest -q`。
6. `git diff --check` 和人工 diff 审查。

Live 合同测试保持默认跳过。本次不注入目标 URL、评论 ID、批准文本或发送开关。

## 完成标准

- 运营手册与代码的新异步架构一致。
- 所有非 live 自动化测试通过，live 测试保持安全跳过。
- `git diff --check` 无错误。
- diff 审查没有发现重复发送、本地浏览器回退或状态机回退风险。
- 项目状态文档记录准确的自动化结果和 live 阻塞，不宣称真实发送成功。

## 非目标

- 不提供真实测试帖子、评论或批准文本。
- 不运行 live selector probe 或真实评论发送。
