# 项目经理对话与独立执行对话协作协议

## 1. 对话结构

项目经理对话是一个持续存在的对话，负责整个项目的全局管理。

每次派发任务时，项目经理都要另外新开一个完全独立的新对话作为执行对话。

执行对话：

- 不是从项目经理对话派生出来的
- 不继承项目经理对话的聊天上下文
- 不需要阅读整套项目文档
- 只接收项目经理生成的自包含任务包
- 在独立分支或 worktree 中工作
- 通过代码提交、测试结果和任务报告交付

结构：

```text
持续存在的项目经理对话
├── 新开的独立执行对话 W1
├── 新开的独立执行对话 W2
├── 新开的独立执行对话 W3
└── 新开的独立执行对话 W4
```

这些执行对话彼此独立，也不共享项目经理的聊天上下文。

## 2. 项目经理职责

- 读取项目全局文档
- 判断当前阶段和任务依赖
- 创建 `orchestration/packets/TXX.md`
- 为任务指定分支或 worktree
- 新开一个独立执行对话
- 把任务包路径和 Worker 提示词交给执行对话
- 审查代码、测试和任务报告
- 决定接受、返工、拒绝或阻塞
- 验收通过后更新全局进度文件
- 合并任务分支

## 3. 独立执行对话职责

- 只读取当前任务包和点名文件
- 只执行一个任务
- 只修改允许范围
- 运行指定测试
- 创建标准任务报告
- 完成后停止

## 4. 为什么不使用派生对话

派生对话会继承大量无关上下文，导致：

- 上下文浪费
- 任务边界模糊
- 不同执行者共享未经文档化的信息
- 难以验证任务包是否完整
- 工具切换和新电脑接手困难

因此协作信息必须写进 Git 仓库，而不是藏在对话继承关系里。

## 5. 并发

项目经理可以同时管理多个独立执行对话。

默认 2 个，稳定后可增加到 3–4 个。

并发前必须检查：

- 任务依赖
- 文件锁
- 公共接口冲突
- 数据库迁移冲突
- 待验收任务数量
- 项目经理是否有能力及时验收

## 6. 交付链路

```text
项目经理创建任务包
→ 新开独立执行对话
→ 执行对话完成代码和测试
→ 提交任务报告
→ 项目经理验收
→ 合并并更新进度
```
