# 决策记录

## D001 第一阶段先做数据发现，不先做自动销售

状态：已确认

原因：

- 需要先积累足够数据，才能验证哪些信号有价值
- 过早做评分、话术和 CRM 会建立在未经验证的假设上

## D002 第一平台选择小红书

状态：已确认

原因：

- 教育和家长内容密度较高
- 帖子、评论和回复具有较强上下文
- 适合作为第一阶段验证平台

## D003 第一阶段保留少量人工预警

状态：已确认

原因：

- 不打断数据积累主线
- 可以尽早获得真实业务反馈
- 为后续评分系统积累标签

## D004 核心任务队列先使用 PostgreSQL

状态：已确认

原因：

- 第一阶段任务规模不需要 Redis 或 Kafka
- 简化部署和维护
- 数据状态更容易审计

## D005 AI 不逐条实时调用

状态：已确认

原因：

- 成本高
- 延迟高
- 大量低价值文本没有必要调用强模型
- 优先规则过滤和批量分析

## D006 内容本体和发现路径分离

状态：已确认

原因：

- 同一内容可能被多个查询发现
- 需要评估查询产出
- 避免重复存储

## D007 事件雷达是核心升级方向

状态：已确认

原因：

- 需求往往由考试、出成绩、暑假和报名等事件触发
- 事件触发式采集比静态关键词更早发现需求

## D008 公开信息采集不是当前主要阻碍

状态：已确认

说明：

- 系统只处理正常公开可见的数据
- 不绕过私密权限
- 不把理论滥用风险扩展成第一阶段的主要开发负担

## D009 不默认使用群控、复杂指纹和大规模账号体系

状态：已确认

原因：

- MVP 首先验证数据价值和采集稳定性
- 复杂设备和账号体系会显著增加成本与维护难度
- 如后续确有必要，再单独评估

## D010 飞书不是主数据库

状态：已确认

原因：

- 飞书适合审核、提醒和人工操作
- 核心数据和关系应存储在 PostgreSQL

## D011 主控会话统一管理子会话

状态：已确认

- 项目需要并行或分阶段执行

规则：

- 子会话只执行单个任务
- 全局状态文件只由主控修改
- 每个子任务必须有简报和报告
- 并行任务使用独立分支或 worktree

## D017 T01 项目骨架依赖基线

状态：已确认

用途：

- FastAPI 和 Uvicorn 用于第一阶段 API 服务与 `/health`
- SQLAlchemy 2.x、Alembic 和 psycopg 用于 PostgreSQL 数据访问与迁移
- Pydantic 用于基础配置和后续数据对象
- pytest 和 httpx 用于测试
- GitHub Actions 用于远端迁移和测试验证

许可证：

- 上述 Python 依赖均为宽松开源许可证生态中的常用依赖，适合当前 MVP 使用；后续新增依赖仍需单独记录。

替代方案：

- Web 框架可选 Flask 或 Django，但 FastAPI 更适合类型标注和 API 原型。
- PostgreSQL 驱动可选 asyncpg，但当前同步 SQLAlchemy + psycopg 更简单。
- 测试框架可选 unittest，但 pytest 更适合项目阶段 0 的快速测试。

原因：

- 与 `docs/ARCHITECTURE.md` 已确认技术栈一致。
- GitHub CI 已在 PostgreSQL service 下通过 Alembic 迁移和测试，补足本机缺 Docker 的验收缺口。

## D018 V0 使用 Playwright 作为真实浏览器采集运行时

状态：已确认

用途：

- 小红书 V0 采集器使用持久化浏览器目录保存人工登录态。
- 真实采集优先读取公开页面、可验证 DOM 和公开网络响应。
- live 测试与普通单元测试分离，避免把 Mock 或离线样本当作真实采集验收。

许可证：

- Playwright 为 Apache-2.0 许可证，适合当前 MVP 使用。

替代方案：

- Selenium 可实现浏览器自动化，但 Playwright 的持久化上下文、网络响应监听、截图和超时控制更适合当前采集与调试需求。
- 直接 HTTP 请求会更轻量，但当前不允许依赖来源不明的私有接口，也不能稳定覆盖登录态、页面结构和公开交互。

约束：

- 不破解验证码。
- 不绕过平台安全机制。
- 不伪造设备指纹。
- 不实现账号池或代理池。
- 不调用来源不明的私有接口。

## D019 飞书作为人工获客工作台和手动中控台

状态：已确认

用途：

- 飞书多维表格用于让非技术用户查看客户线索、原始证据、AI 推荐原因和人工确认状态。
- 飞书 `系统控制台` 用于人工发出一次性指令，例如找新客户、刷新客户表、同步确认结果和查看系统状态。
- 系统只在本机执行 `run-control-panel-once` 时读取一次控制台，不自动后台轮询，不偷偷执行。

页面文案规则：

- 面向用户的飞书字段不使用 pipeline、worker、JSON、CLI、record id、token 等技术词。
- 用户只看到 `我要做什么`、`开始执行`、`现在状态`、`要找什么`、`最多看多少条`、`结果`、`哪里出错了` 等普通表达。
- 技术命令、参数和内部 ID 只写在代码、README 和交接文档里。

安全规则：

- 控制台只允许白名单动作，不能把飞书字段拼成任意 shell 命令执行。
- 当前白名单是：`找新客户`、`重新整理客户`、`刷新客户表`、`同步确认结果`、`查看系统状态`。
- 每次执行完成后必须把 `开始执行` 改回 `否`，避免重复执行。

原因：

- 用户明确要求不要自动跑，必须人工发指令后再运行。
- 飞书 Base 比 Excel 的价值在于能作为审核台、状态机和人工触发面板。
- 先采用一次性命令比实时 webhook 更安全、更容易验证，也不依赖公网回调地址。

## D020 飞书话术审批和小红书真实发送必须分离

状态：已确认

规则：

- 飞书话术审批卡里的“发送”按钮只表示人工确认话术可以发送。
- 点击后系统写入 `approved_to_send`、最终话术、审核人、审核时间和飞书消息信息。
- 飞书回调线程不得直接打开小红书浏览器发送私信。
- 真实小红书发送只能由独立入口或 worker 处理，并记录 `sending`、`sent`、`failed`、`last_error` 和 `attempt_count`。
- 当前本机浏览器/网络环境无法稳定打开小红书私信页，且用户要求不要改 Clash；在该问题解决前，不进行真实小红书自动发送。

原因：

- 飞书回调必须快速返回，避免按钮卡住或回调超时。
- 小红书浏览器自动化受登录态、VPN、页面变化和风控影响，不应阻塞飞书审核流程。
- 审批入库和实际发送分离后，失败能被明确记录和重试，不会把未发送伪造成已发送。

## D021 评论回复真实验收必须先探测再逐条批准

状态：已确认

规则：

- 评论回复自动化实现完成不等于真实小红书发送验收完成。
- 真实验收必须使用明确准备的单条目标，先执行不提交的 selector probe，再获得飞书人工对最终文本和本次发送的明确批准。
- `result_unknown` 一律人工核对，禁止盲目重试；飞书/Base 同步重试不得触发平台重发。
- 运营、恢复、卡片认领和 Base 配置统一以 `docs/COMMENT_REPLY_OPERATIONS.md` 为准。

## D022 评论回复通过持久任务复用远程 Windows CDP

状态：已确认

规则：

- 飞书“确认发送”回调只完成审批入库并创建 `comment_reply_send` 持久任务，立即返回 `accepted`；不得在回调请求内运行 Playwright。
- 评论发送 Worker 必须读取 `COMMENT_REPLY_BROWSER_MODE=remote_cdp` 和 `COMMENT_REPLY_CDP_URL`，连接远程 Windows Chrome 的现有 context，不在 Mac 启动本地浏览器。

- `approved_to_send` 表示已人工批准、等待独立任务执行；客户跟进表映射为“评论已批准，等待发送”。

## 2026-07-16：V19 由主控监督独立执行对话连续交付

- 用户明确要求当前对话只担任主控，不直接实现子任务代码。
- V19-02 至 V19-07 每个子任务必须新开独立执行对话，并在独立分支/worktree 中工作；只创建分支不等于创建执行对话。
- 主控负责自包含任务包、任务难度和推理强度分配、文件锁、过程监督、代码与真实证据验收、返工、合并、发布和用户汇报。
- 各执行对话只执行一个任务，不修改全局进度文件，完成报告后停止等待主控验收。
- 主控可在用户已批准的 V19 总发布范围内连续派发后续任务，但仍不得绕过不可逆操作确认或真实公开回复的逐条最终批准。

## 2026-07-16：客户 CRM 身份与阶段分离

- 是否属于正式客户统一由 `Lead.status == qualified` 判断；`crm_stage` 只表示正式客户所处业务阶段，不承担客户身份判定。
- 正式客户进入 `invalid/无效` 阶段后仍保留在 CRM、Operator API 和同步范围内；非正式 candidate、watch 或 ignored Lead 不得进入客户投影。
- Base 白名单 pull 必须在创建或收养 `FeishuBitableRecord` mapping 前验证正式客户身份，禁止通过手工填写后端 ID 和同步版本改写候选或生成客户审计事实。

## 2026-07-16：V19 写接口使用持久请求幂等事实

- 非空 `idempotency_key` 和实体唯一约束不能替代请求幂等；同一 key 重试必须返回首次稳定结果，不能继续产生新的队列副作用。
- V19-03 的 report rebuild、prepare review queue、continue review queue 共用 `ReviewQueueOperation` 持久事实；只保存 key 的 SHA-256、规范化请求、首次结果、操作域、业务日和审计时间，不保存 key 原文。
- 同 key 但参数、业务日、Run 或操作域不同必须明确拒绝并返回安全错误；不得悄悄复用或泄漏 key、密钥、堆栈。
- 已经执行的 `0019_review_queue` 不允许改写历史，因此使用独立 `0020_review_queue_idempotency` 迁移，避免 Alembic revision 漂移。
